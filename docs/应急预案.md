# 宁夏CRM作战地图 — 应急预案

> 本文档定义系统故障的分级标准、响应流程和处置方案，确保在发生故障时能够快速恢复服务，最小化业务影响。

---

## 一、故障分级

### 1.1 分级定义

| 级别 | 名称 | 定义 | 业务影响 | 示例 |
|------|------|------|----------|------|
| **P0** | 紧急 | 系统完全不可用，核心业务中断 | 所有用户无法使用 | 数据库崩溃、服务完全宕机、数据丢失 |
| **P1** | 高 | 核心功能不可用，严重影响业务 | 主要业务流程受阻 | 登录失败、关键API报错、支付中断 |
| **P2** | 中 | 非核心功能异常，业务可降级运行 | 部分功能受限 | 导出失败、报表加载慢、消息延迟 |
| **P3** | 低 | 轻微异常，不影响主业务流程 | 用户体验下降 | 界面显示异常、日志警告、性能轻微下降 |

### 1.2 分级判定矩阵

| 现象 | 用户范围 | 功能影响 | 数据风险 | 级别 |
|------|----------|----------|----------|------|
| 系统完全无法访问 | 全部 | 全部 | 无 | P0 |
| 数据库无法连接 | 全部 | 读写全部 | 高 | P0 |
| 登录/认证失败 | 全部 | 全部 | 无 | P1 |
| 核心模块报错 | 部分 | 1-2个模块 | 无 | P1 |
| 导出/报表功能异常 | 部分 | 非核心功能 | 无 | P2 |
| 页面加载慢(>5s) | 部分 | 全部 | 无 | P2 |
| 界面显示问题 | 个别 | 视觉 | 无 | P3 |
| 日志警告增加 | 无 | 无 | 无 | P3 |

## 二、响应时间要求

### 2.1 响应时间SLA

| 级别 | 首次响应时间 | 故障定位时间 | 临时恢复时间 | 完全恢复时间 | 复盘报告时间 |
|------|-------------|-------------|-------------|-------------|-------------|
| **P0** | ≤ 15分钟 | ≤ 30分钟 | ≤ 1小时 | ≤ 4小时 | 24小时内 |
| **P1** | ≤ 1小时 | ≤ 2小时 | ≤ 4小时 | ≤ 8小时 | 48小时内 |
| **P2** | ≤ 4小时 | ≤ 8小时 | ≤ 24小时 | ≤ 48小时 | 72小时内 |
| **P3** | ≤ 24小时 | ≤ 48小时 | ≤ 72小时 | 下次发版 | 下次发版前 |

### 2.2 响应时间定义

| 阶段 | 定义 | 说明 |
|------|------|------|
| 首次响应 | 告警通知到运维人员开始处理 | 从Alertmanager发告警开始计时 |
| 故障定位 | 确定故障根因或影响范围 | 可通过日志、监控、排查确认 |
| 临时恢复 | 业务恢复可用，可能有降级 | 如回滚、重启、切换备用方案 |
| 完全恢复 | 系统恢复正常状态，无降级 | 根因修复，无遗留问题 |
| 复盘报告 | 输出故障分析报告 | 包含根因、修复过程、改进措施 |

### 3.2 各阶段操作清单

#### 阶段1：告警接收（0-5分钟）

- [ ] 确认收到告警通知（企微/钉钉/短信/电话）
- [ ] 在值班群回复"收到告警，正在处理"
- [ ] 打开监控面板确认告警详情
- [ ] 记录故障开始时间

#### 阶段2：初步判断（5-15分钟）

- [ ] 检查系统整体状态（Grafana概览面板）
- [ ] 快速查看最近变更（部署记录/配置变更）
- [ ] 检查相关服务日志（Loki或docker logs）
- [ ] 判断故障类型和影响范围
- [ ] 确定故障级别（P0/P1/P2/P3）

#### 阶段3：处置响应（根据级别）

- [ ] P0：立即升级通知技术主管+项目经理，启动紧急响应
- [ ] P1：通知技术主管，视情况升级
- [ ] P2/P3：记录问题，排入处理队列
- [ ] 执行对应故障处理方案（见第四章）
- [ ] 每30分钟更新处理进展到值班群

#### 阶段4：验证恢复

- [ ] 确认服务恢复正常（调用健康检查接口）
- [ ] 确认核心业务功能可用（执行验证用例）
- [ ] 检查数据一致性（关键表抽样）
- [ ] 持续观察15分钟确认稳定
- [ ] 记录故障恢复时间

#### 阶段5：复盘总结

- [ ] 收集故障时间线
- [ ] 分析根因（5 Whys）
- [ ] 输出改进措施（预防+检测+响应）
- [ ] 更新应急预案（如适用）
- [ ] 安排技术分享会议

## 四、常见故障处理

### 4.1 数据库连不上

#### 症状
- 后端日志大量 `connection refused` 或 `timeout`
- 用户反馈页面加载失败或数据无法保存
- Grafana 显示 `CRMDatabaseDown` 告警

#### 排查步骤

1. 检查数据库容器状态
   ```bash
   sudo docker ps | grep crm-postgres
   # 若状态为 Exited，查看日志
   sudo docker logs --tail 50 crm-postgres
   ```

2. 检查数据库连接数
   ```bash
   sudo docker exec -it crm-postgres psql -U crm_user -d crm_production -c "SELECT count(*) FROM pg_stat_activity;"
   ```

3. 检查服务器资源
   ```bash
   # 磁盘空间（PostgreSQL崩溃常见原因）
   df -h
   # 内存使用
   free -h
   ```

4. 检查数据库进程
   ```bash
   sudo docker top crm-postgres
   ```

#### 处置方案

| 场景 | 处置方法 | 预计恢复时间 |
|------|----------|-------------|
| 容器已退出 | `docker compose restart postgres` | 1-2分钟 |
| 磁盘满 | 清理日志/临时文件，或扩容磁盘 | 5-10分钟 |
| 连接数满 | 重启应用释放连接，或调整max_connections | 2-5分钟 |
| 数据损坏 | 使用最新备份恢复，见备份恢复手册 | 30-60分钟 |
| 主库宕机 | 切换到只读模式或启用备用实例 | 10-30分钟 |

> **警告**：数据库恢复操作前，务必先进行当前状态备份，避免二次损坏！

### 4.2 后端服务挂掉

#### 症状
- 所有API返回502/503
- 前端页面显示"服务不可用"
- `CRMServiceDown` 告警触发

#### 排查步骤

1. 检查后端容器状态
   ```bash
   sudo docker ps | grep crm-backend
   sudo docker logs --tail 100 crm-backend
   ```

2. 检查内存是否溢出（OOM）
   ```bash
   sudo dmesg | grep -i "out of memory"
   sudo docker inspect crm-backend | grep OOMKilled
   ```

3. 检查端口占用
   ```bash
   sudo netstat -tlnp | grep 3001
   ```

4. 检查依赖服务（数据库/Redis/MinIO）
   ```bash
   sudo docker compose -f docker-compose.prod.yml ps
   ```

#### 处置方案

| 场景 | 处置方法 | 预计恢复时间 |
|------|----------|-------------|
| 容器退出 | `docker compose restart backend` | 1-2分钟 |
| OOM Killed | 增加容器内存限制，或检查内存泄漏 | 2-5分钟 |
| 依赖服务未就绪 | 按依赖顺序重启服务（Postgres→Redis→MinIO→Backend） | 3-5分钟 |
| 应用错误循环 | 回滚到上一版本镜像 | 5-10分钟 |
| 端口冲突 | 检查并释放占用端口，或修改端口映射 | 2-5分钟 |

### 4.3 内存溢出（OOM）

#### 症状
- 系统响应极慢或无响应
- 服务进程被系统杀死
- `CRMHighMemoryUsage` 告警触发
- 系统日志出现 `Out of memory: Kill process`

#### 排查步骤

1. 查看当前内存使用
   ```bash
   free -h
   cat /proc/meminfo | grep Available
   ```

2. 查看各进程内存占用
   ```bash
   ps aux --sort=-%mem | head -20
   ```

3. 查看容器内存限制
   ```bash
   sudo docker stats --no-stream
   ```

4. 查看JavaScript/Node.js内存（如适用）
   ```bash
   # 进入后端容器
   sudo docker exec -it crm-backend sh
   # 查看Node.js内存
   node -e "console.log((require('v8').getHeapStatistics().used_heap_size / 1024 / 1024).toFixed(2) + ' MB')"
   ```

#### 处置方案

| 场景 | 处置方法 | 预计恢复时间 |
|------|----------|-------------|
| 内存泄漏 | 重启服务容器，增加内存限制 | 1-2分钟 |
| 并发请求过高 | 增加服务器资源或启用限流 | 5-10分钟 |
| 大数据查询 | 优化查询，增加分页/缓存 | 需开发介入 |
| 内存不足 | 临时增加swap分区，长期扩容内存 | 10-30分钟 |

> **临时增加Swap（应急）**：
> ```bash
> sudo fallocate -l 2G /swapfile
> sudo chmod 600 /swapfile
> sudo mkswap /swapfile
> sudo swapon /swapfile
> ```

### 4.4 磁盘空间不足

#### 症状
- `CRMHighDiskUsage` 告警触发
- 数据库写入失败
- 文件上传失败

#### 排查步骤

```bash
# 查看磁盘使用

df -h

# 查看大文件/目录
sudo du -sh /var/lib/docker/* 2>/dev/null | sort -rh | head -10
sudo du -sh /var/log/* 2>/dev/null | sort -rh | head -10

# 查看Docker占用
sudo docker system df -v

# 查看日志大小
sudo find /var/log -type f -exec du -sh {} + 2>/dev/null | sort -rh | head -10
```

#### 处置方案

| 场景 | 处置方法 | 预计恢复时间 |
|------|----------|-------------|
| 日志过大 | 清理旧日志，配置logrotate | 5分钟 |
| Docker镜像/卷过大 | `docker system prune -a` 和清理旧卷 | 10-15分钟 |
| 数据库膨胀 | 清理过期数据，或VACUUM | 需DBA介入 |
| 数据盘满 | 扩容云盘或迁移数据 | 30-60分钟 |

> **清理Docker（谨慎执行）**：
> ```bash
> # 清理未使用容器、网络、镜像、缓存
> sudo docker system prune -a --volumes
> # 仅清理已停止的容器和未使用的镜像
> sudo docker container prune -f
> sudo docker image prune -a -f
> ```

### 4.5 网络异常

#### 症状
- 部分用户无法访问，部分正常
- 请求超时或间歇性失败
- 外部API/ webhook 调用失败

#### 排查步骤

```bash
# 测试外网连通性
ping -c 4 8.8.8.8
ping -c 4 crm.example.com

# 测试端口连通性
curl -v telnet://crm.example.com:443

# 检查Nginx状态
sudo systemctl status nginx
sudo nginx -t

# 检查防火墙
sudo ufw status
sudo iptables -L -n | grep DROP
```

#### 处置方案

| 场景 | 处置方法 | 预计恢复时间 |
|------|----------|-------------|
| DNS解析失败 | 检查DNS配置，切换备用DNS | 5-10分钟 |
| 防火墙拦截 | 检查并调整UFW/iptables规则 | 5分钟 |
| Nginx配置错误 | 回滚配置，`nginx -t`验证后reload | 5分钟 |
| 带宽超限 | 联系运营商扩容，或启用CDN | 需商务介入 |
| DDoS攻击 | 启用WAF/高防IP，联系云服务商 | 30-60分钟 |

## 五、数据恢复流程

### 5.1 数据恢复决策树

```
                    ┌─────────────┐
                    │  数据丢失？  │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │ 是          │ 否          │
              ▼            ▼            ▼
        ┌─────────┐  ┌─────────┐  ┌─────────┐
        │数据丢失范围│  │业务不可用│  │数据损坏 │
        │评估      │  │服务故障  │  │需修复   │
        └────┬────┘  └────┬────┘  └────┬────┘
             │            │            │
             ▼            ▼            ▼
        ┌─────────┐  ┌─────────┐  ┌─────────┐
        │全量恢复  │  │服务恢复  │  │定点恢复  │
        │最新备份  │  │流程     │  │到特定时间│
        └─────────┘  └─────────┘  └─────────┘
```

### 5.2 数据库恢复流程

1. **停止写入服务**
   ```bash
   sudo docker compose -f docker-compose.prod.yml stop backend
   ```

2. **备份当前状态（即使已损坏）**
   ```bash
   sudo docker exec -i crm-postgres pg_dump -U crm_user crm_production > /backups/emergency_backup_$(date +%Y%m%d_%H%M%S).sql
   ```

3. **选择恢复方案**

   | 恢复方案 | 适用场景 | 命令 |
   |----------|----------|------|
   | 完整恢复 | 数据库完全损坏 | `psql -U crm_user -d crm_production < db_backup_YYYYMMDD.sql` |
   | 单表恢复 | 特定表数据丢失 | 提取备份中的对应表数据，INSERT恢复 |
   | 时间点恢复 | 需要回退到特定时间 | 使用WAL归档（如已配置） |

4. **执行恢复**
   ```bash
   # 完整恢复示例
   sudo docker exec -i crm-postgres psql -U crm_user -d crm_production < /backups/db_backup_20260101_040000.sql
   ```

5. **验证数据完整性**
   ```bash
   sudo docker exec -it crm-postgres psql -U crm_user -d crm_production -c "SELECT COUNT(*) FROM \"User\";"
   sudo docker exec -it crm-postgres psql -U crm_user -d crm_production -c "SELECT COUNT(*) FROM \"Project\";"
   sudo docker exec -it crm-postgres psql -U crm_user -d crm_production -c "SELECT COUNT(*) FROM \"Customer\";"
   ```

6. **恢复服务**
   ```bash
   sudo docker compose -f docker-compose.prod.yml start backend
   ```

> **警告**：数据恢复操作不可逆，执行前务必双重确认备份文件和恢复目标！建议在测试环境先演练恢复流程。
