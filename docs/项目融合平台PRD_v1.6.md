## 1. 文档目的

本文档用于定义“项目管理工具 + 商机雷达”融合方案从单点 M23 MVP 向综合业务平台演进后的产品方案。v1.5 在 v1.4 基础上融合《宁夏区域CRM作战地图_PRD_v1.1_含SOP》的客户关系管理能力，补充区域客户作战地图、权力地图、客户树、拜访互动、关系健康度和数据质量管理。**v1.6 进一步完整融合 CRM PRD v1.2 新增内容：前端交互设计规范、API 接口规范和前端 23 个页面的交互映射清单，实现从业务需求到前端实现的完整链条。**

v1.5 不再只描述一个轻量商机雷达，而是定义一套面向信息化服务公司的业务工作台。根据当前 Demo，产品主导航收敛为六个一级模块：作战台、项目交付、运维管理、商机营销、客户资产、知识分享。它仍然保留第一期 MVP 的收敛原则：先让项目、运维、客户、关系和商机数据在一个系统里形成闭环，再逐步扩展完整合同、报价、审批、AI 和自动化变动抓取能力。

原 M23 MVP 的最小闭环仍然成立：

`客户档案 -> 关键资产 -> 运维记录 -> 规则识别 -> 商机信号 -> 负责人跟进 -> 结果回填`

v1.5 扩展后的主闭环为：

`项目立项 -> 任务交付 -> 运维巡检 -> 记录沉淀 -> 知识复用 -> 商机识别 -> 区域作战地图 -> 客户拜访/关系维护 -> 培训/方案交付 -> 续约/扩容/服务升级`

本文档将作为研发排期、测试验收、试点上线和业务评审的依据。

## 2. 背景与问题

信息化公司在项目交付、运维服务、客户经营之间通常存在明显断层。运维人员最接近客户现场，最早知道系统性能、资产到期、安全隐患、客户抱怨和新需求，但这些信息如果只停留在工单、微信群、个人笔记或口头沟通里，就很难转化为可跟进的续约、扩容、安全加固、系统改造和服务升级机会。

原 v1.2 方案已完整描述平台远景，但范围过大，第一期容易被完整项目管理、工单、变更、AI、大屏、项目集等能力拖住。v1.3 将 M23 独立收敛为轻量 MVP，先验证“运维记录能否稳定产生真实商机”。

## 3. 产品定位

### 3.1 一句话定位

面向信息化服务公司的项目交付、运维服务、客户经营和培训交付一体化平台：把项目过程、运维记录、巡检事实、知识经验、客户培训和商机线索沉淀到同一个业务闭环中，帮助团队把服务交付转化为可复用知识和可跟进增长机会。

### 3.2 六大工作模块

| 一级模块 | 核心问题 | 产品能力 | 业务结果 |
|---|---|---|---|
| 作战台 | 团队每天不知道优先处理什么 | 今日概览、待处理、经营报表、数据导出、团队设置 | 一屏聚合项目、运维、客户、商机和提醒 |
| 项目交付 | 项目前期调研、售前方案、合同评审、实施交接、采购、需求、变更、测试、验收、运维移交分散在表格和群消息中 | 项目看板、任务排期、交付简报、全流程节点、会议纪要、风险问题、交付物 | 项目从售前到运维可视、责任清晰、交付经验可复用 |
| 运维管理 | 运维经验只留在个人、工单或聊天记录里 | 运维记录、巡检计划、巡检记录、资产台账、知识沉淀、规则配置 | 形成公司级运维知识资产，并从现场事实识别机会 |
| 商机营销 | 客户现场信号无法转成续约、扩容、服务升级机会 | 商机池、营销跟进、拜访计划、来源证据链和下次动作 | 提升续约、扩容、安全加固和服务升级转化 |
| 客户资产 | 客户资料、区域分布、关键人关系和互动记录不可交接 | 客户档案、区域地图、权力地图、客户树、关系网络、可视化协作、数据质量和互动历史 | 客户关系从个人经验转为组织资产 |
| 知识分享 | 培训材料、客户方案、交付文档无法系统复用 | 培训计划、方案素材、FAQ、知识引用和客户简报素材 | 降低培训交付成本，提高客户使用深度 |

### 3.3 参考产品启发

本产品不直接复制某一类工具，而是组合学习以下产品模式：

| 参考方向 | 可借鉴点 | 本产品取舍 |
|---|---|---|
| 金蝶类项目管理/企业协同工具 | 项目全生命周期、任务责任、交付物、风险预警、业财和合同衔接 | 先做项目台账、任务、里程碑、风险和交付物，不在 MVP 做复杂预算、成本、回款和 ERP 深集成 |
| 上上签类电子签约/合同流程工具 | 合同全生命周期、审批签署、存证、到期提醒、履约节点 | 先借鉴“合同/维保到期提醒、存证和流程留痕”，不在 MVP 做电子签章、法律存证和合同审批 |
| 飞书项目/飞书知识库、Notion、Confluence 类协同知识产品 | 项目协同、结构化文档、知识库、模板复用、跨团队协作 | 先做轻量知识条目、方案模板、项目复盘，不做复杂文档权限和多人编辑器 |
| 禅道、Tapd、Jira 类项目/任务工具 | 任务流转、缺陷问题、迭代看板、负责人和状态 | 保留任务和风险问题管理，不把产品做成研发缺陷管理工具 |
| 客户成功/CRM 类工具 | 客户档案、跟进记录、健康标签、续约和扩容机会 | 只做服务型客户经营，不替代完整 CRM 销售漏斗 |
| 纷享销客、Salesforce、Pipedrive、Monday CRM、Zoho CRM | 权力地图、客户树、销售管道、可视化协作、数据质量和健康度模型 | 先做区域客户地图、关键人关系链、拜访互动和轻量健康度，不在 MVP 做完整销售自动化 |

### 3.4 产品边界

v1.5 的产品定位是“服务交付与客户经营一体化工作台”。M23 商机雷达是其中的客户经营子模块，区域 CRM 作战地图是客户经营的组织化载体。

它不做：

- 不替代完整 ERP、财务、采购和成本核算系统。
- 不替代完整 CRM 销售流程和报价回款流程。
- 不替代复杂 ITSM 工单系统。
- 不承担合同回款、报价审批和电子签章法律能力。
- 不要求运维人员承担销售职责。
- 不在 MVP 阶段做 AI 销售顾问、AI 培训讲师、复杂知识图谱或外部人事变动自动抓取。

它只做：

- 汇集项目、客户、资产、运维记录、巡检计划、巡检记录和培训记录。
- 用轻量任务和里程碑管理项目交付过程。
- 将运维记录沉淀为可检索、可复用的知识和方案。
- 用规则识别高频巡检商机信号，并推动跟进。
- 用区域 CRM 作战地图呈现客户分布、关系链、拜访计划和重点客户状态。
- 记录客户培训、方案交付和客户能力建设过程。
- 为后续 CRM、工单、合同、电子签、AI、客户健康度和业财集成打基础。

## 4. MVP 目标与非目标

### 4.1 业务目标

在 8-10 周内上线一个可试点的 Web MVP，支持 10-20 个真实客户、3-5 个进行中项目、2-3 名运维工程师、1-2 名项目经理、1-2 名客户经理/CSM、1 名培训负责人、1 名区域负责人和 1 名主管完成试运行。

第一期目标不是做“大而全项目管理平台”，而是验证四个闭环是否能跑通：

1. 项目任务能否沉淀为可复盘的交付过程。
2. 运维记录能否稳定沉淀为知识和方案。
3. 巡检记录能否稳定产生真实商机。
4. 客户培训能否关联客户、项目、方案和后续服务机会。
5. 客户关系、关键人和拜访互动能否沉淀为可交接的组织资产。

### 4.2 验证假设

| 假设 | 验证方式 | 通过标准 |
|---|---|---|
| 运维现场信息能稳定产生商机线索 | 持续录入运维记录并生成信号 | 每月不少于 5 条可跟进商机 |
| 运维人员愿意低摩擦补充记录 | 快速记录入口、模板点选、非强制长文本 | 补充记录完成率 > 50% |
| 项目经理/客户经理愿意处理信号 | 商机池、提醒、分配、状态流转 | P0/P1 信号 48h 内响应率 > 80% |
| 规则识别不会明显透支信任 | 误报反馈、人工确认 | P0/P1 误报率 < 15% |
| 到期资产能带来明确续约动作 | 关键资产录入、到期提醒 | 关键资产覆盖率 > 70% |
| 项目过程能被项目经理持续维护 | 项目台账、任务、里程碑、风险记录 | 试点项目任务更新率 > 80% |
| 运维知识能被复用 | 运维记录转知识、知识检索、方案引用 | 每月至少沉淀 10 条有效知识 |
| 客户培训能促进后续服务 | 培训计划、培训记录、方案推荐 | 每个试点客户至少 1 条培训/方案记录 |
| 区域客户关系能被组织化管理 | 重点客户关系链、拜访记录、数据质量评分 | 重点客户关系链覆盖率 >= 80% |
| 客户拜访动作能被复盘 | 拜访计划、互动记录、下次动作 | 拜访后 48h 记录回填率 >= 80% |

### 4.3 非目标

MVP 阶段不做以下能力：

- 完整 CRM 销售流程。
- 外部政府网站、脉脉、LinkedIn 的自动变动抓取。
- 完整工单系统。
- 合同回款、报价、审批。
- 电子签章、合同法律存证。
- 复杂项目成本、预算、回款和业财一体化。
- 多项目集资源池和复杂排期算法。
- 在线考试系统和完整 LMS 学习平台。
- AI 销售顾问。
- 客户健康度复杂评分。
- 复杂图数据库关系推理和自动客户树发现。
- 大屏。
- 移动 App。
- 项目集协同。
- 规则版本回滚。
- 复杂权限审批流。
- M12 变更自动创建。
- M16 完整工单关闭弹窗。

## 5. 用户角色与职责

### 5.1 角色定义

| 角色 | 核心职责 | MVP 中的动作 |
|---|---|---|
| 运维工程师 | 记录现场事实，不承担销售职责 | 新增运维记录、选择问题模板、补充客户反馈 |
| 项目经理 | 判断交付期客户风险与机会 | 查看信号、确认/放弃、创建跟进 |
| 客户经理/CSM | 负责客户沟通与商机推进 | 接收分配、记录跟进、填写意向 |
| 区域负责人 | 负责区域客户覆盖、重点客户推进和跨团队协同 | 查看区域作战地图、识别空白客户、复盘拜访和机会 |
| 主管 | 分配无负责人信号、处理超时 | 查看待分配、查看超时、调整负责人 |
| 产品/运营管理员 | 维护规则和基础配置 | 配置关键词、查看误报、维护字典 |

### 5.2 负责人分配规则

| 场景 | 默认负责人 | 兜底 |
|---|---|---|
| 项目执行期客户 | 项目经理 | 部门主管 |
| 运维服务期客户 | 客户经理/CSM | 部门主管 |
| 纯维保客户 | 服务合同负责人 | 客户经理 |
| 无负责人客户 | 进入主管分配池 | 24h 未分配升级 |
| 预计金额 >= 10 万 | 同步客户经理/销售负责人 | 主管确认 |

## 7. 核心业务流程

### 7.1 主流程

1. 运维工程师在快速记录入口新增一条运维记录。
2. 系统保存客户、资产、问题模板、文本描述和客户反馈。
3. 规则引擎根据到期时间、关键词、重复故障等条件识别信号。
4. 系统生成商机信号，写入商机池。
5. 系统按负责人分配规则设置 owner；无负责人进入主管分配池。
6. 负责人在商机池中点击“开始跟进”，填写下次跟进时间。
7. 负责人根据沟通结果选择“确认商机 / 放弃 / 暂缓 / 误报”。
8. 已确认商机进入后续真实业务动作：报价、变更、立项或客户拜访。
9. 系统持续记录状态变化、跟进记录和通知日志。

### 7.2 运维记录原则

运维端只记录事实，不要求判断商机。界面文案应避免“销售”“商机转化”等表达，优先使用：

- 现场补充
- 客户反馈
- 风险线索
- 系统现象
- 后续建议

### 7.3 项目管理闭环

1. 项目经理创建项目，选择客户、项目阶段、项目成员和计划周期。
2. 项目经理拆分里程碑和任务，明确负责人、截止时间和交付物。
3. 成员按任务更新进度，上传方案、计划、会议纪要、验收材料等交付物。
4. 项目风险或问题出现时，项目经理登记风险、影响范围、处理人和截止时间。
5. 项目进入上线、试运行、验收等节点时，系统提醒负责人补齐交付物。
6. 项目结束后触发复盘流程，沉淀项目经验、问题清单、可复用方案和培训材料。
7. 复盘内容可进入知识库，并可关联到客户、行业、产品模块和后续项目。

### 7.4 运维知识沉淀闭环

1. 运维工程师新增运维记录或巡检记录。
2. 系统保存客户、资产、问题模板、处理过程、客户反馈和结果。
3. 记录关闭后，如果满足“可复用”条件，系统提醒沉淀知识。
4. 运维人员或主管将记录整理为知识条目：适用场景、处理步骤、注意事项、附件。
5. 知识条目通过标签、客户、行业、资产、问题模板进行分类。
6. 后续新增运维记录、巡检记录或培训问题时，系统可检索并推荐相关知识。
7. 如果知识被多次引用或反馈无效，进入知识维护提醒。

### 7.5 客户培训与方案闭环

1. 项目经理、客户经理或培训负责人为客户创建培训计划。
2. 培训计划关联客户、项目、产品模块、方案材料和讲师。
3. 培训后记录参训人员、客户问题、培训反馈和后续动作。
4. 客户问题可沉淀为 FAQ 或知识条目。
5. 如果培训中出现新需求、服务升级、扩容、安全加固等信号，系统生成商机信号或人工补录商机。
6. 培训记录沉淀到客户详情页，作为后续续约、复购、项目复盘和客户简报素材。

### 7.6 区域 CRM 作战地图闭环

1. 客户经理或管理员导入客户基础数据，补充地市、行业、等级、负责人和服务状态。
2. 系统按地市、行业、客户等级、负责人和风险标签生成区域作战地图。
3. 客户经理为重点客户维护关键人关系链，至少包含决策人、影响人、使用人或信息人。
4. 客户经理制定拜访或互动计划，明确拜访目的、对象、材料和下一步动作。
5. 拜访、电话、微信、会议、培训等互动完成后，48 小时内回填互动记录。
6. 系统根据最近互动、关系链完整度、项目/商机风险和数据质量更新客户状态。
7. 区域负责人按月复盘地市覆盖、重点客户健康度、拜访达成、商机推进和数据质量。

### 8.2 作战台

目标：让项目经理、运维负责人和客户经理一进入系统就知道今天要处理什么，以及哪些项目/客户/商机需要升级处理。

模块：

- 项目概览：进行中项目、延期任务、待验收里程碑、项目风险。
- 今日待跟进：下次跟进时间 <= 今天的商机。
- 高优先级信号：P0/P1 且未处理。
- 即将到期：30 天内维保/证书到期。
- 高风险客户：有重复故障、安全风险或合同到期。
- 最新运维记录：最近产生线索的记录。
- 知识待沉淀：已关闭但未复盘的故障/项目。
- 培训待办：本周客户培训、待补课、待反馈。
- 区域概览：按地市、行业、客户等级和负责人展示客户数量、风险客户、待拜访客户和有效商机。
- 关系待维护：重点客户缺少决策人、90 天未互动、数据质量低于阈值。

关键动作：

- 新增项目。
- 新增任务。
- 新增运维记录。
- 新增巡检计划。
- 新增巡检记录。
- 开始跟进。
- 分配负责人。
- 标记已确认。
- 标记暂缓。
- 标记误报。
- 生成客户简报。
- 新建培训计划。
- 打开区域作战地图。
- 新增拜访计划。
- 维护关键人关系链。

### 8.3 项目交付

目标：参考项目管理工具的基本逻辑，把项目从前期调研、方案售前、合同评审、售前到实施交接、采购、需求调研、实施沟通、变更、测试、初验、验收、运维移交的全过程，从“群消息 + 表格”收敛为可追踪、可复盘、可沉淀的交付对象。

模块：

- 项目列表：项目名称、客户、阶段、项目经理、计划时间、健康状态。
- 项目分类：支持按项目金额、金额分层、软件项目、系统集成、数字化转型、运维服务、培训/专项服务等维度筛选和统计。
- 项目详情：基本信息、成员、全流程节点、任务、采购到货、设备安装调试、需求与变更、测试验收、风险、交付物、客户培训、复盘。
- 任务看板：按状态查看待办、进行中、待验收、已完成。
- 全流程节点：默认可按项目类型生成，也允许项目经理手动新增、删除、排序、改名和设置强制确认。
- 会议纪要：支持每次客户交流、项目例会、阶段汇报、验收沟通通过录音生成纪要。
- 售前到实施交接：支持沉淀前期调研、方案沟通、合同评审、承诺事项、风险边界、客户干系人和交付范围。
- 采购与资源准备：支持登记采购清单、供应商、设备到货计划、设备到货确认、授权许可、依赖资源和阻塞风险。
- 需求与变更：支持需求调研、方案功能确认、需求确认、变更申请、影响评估、客户确认和版本留痕。
- 设备安装调试：支持设备上架、环境准备、安装部署、参数配置、联调调试、问题整改和现场确认。
- 测试、培训与验收：支持测试计划、问题整改、客户培训、初验、终验、验收材料和客户确认。
- 运维移交：支持运维资料、资产台账、巡检计划、知识条目、培训记录和服务责任人移交。
- 风险问题：风险等级、影响范围、处理人、截止时间、处理状态。
- 交付物：方案文档、计划、会议纪要、验收材料、培训材料。

项目分类规则：

- 按项目金额：可配置金额区间，例如 10 万以下、10-50 万、50-100 万、100 万以上。
- 按项目类型：软件、系统集成、数字化转型、运维服务、培训、专项服务、其他。
- 按交付复杂度：轻量、标准、复杂，可由项目经理手动选择。
- 按客户属性：重点客户、一般客户、新客户、续约客户。

流程节点字段：

- 节点名称：如前期调研、方案沟通、合同评审、售前交接、项目启动、项目采购、需求调研、设备到货确认、方案功能确认、设备安装调试、项目沟通交付、变更管理、联调测试、客户培训、初验、终验、运维移交、复盘。
- 节点阶段：售前、合同、交接、采购、到货、需求、实施、沟通、变更、测试、培训、初验、验收、运维、复盘。
- 节点负责人：节点 Owner，可不同于项目经理。
- 节点状态：未开始、进行中、待确认、已确认、延期、阻塞、跳过。
- 计划时间：计划开始、计划完成。
- 实际时间：实际开始、实际完成。
- 确认要求：是否必须确认、确认人、确认时间、确认意见。
- 输出产物：方案、计划、纪要、配置清单、测试报告、培训材料、验收单、复盘报告等。
- 完成条件：必填字段、必须上传的产物、必须完成的任务或客户确认。

会议纪要能力：

- 纪要中的客户侧参会人可同步到客户资产模块的关键人关系链；若客户关键人不存在，进入“待确认关键人”列表。
- 纪要中识别出的客户反馈、风险、需求变更、续约/扩容/安全加固等信息，可转为项目风险、任务、商机信号或客户互动记录。
- 项目干系人关系与客户管理关系保持同步：项目里新增或确认的客户侧干系人，应同步到客户档案、权力地图和可视化协作视图；客户侧关键人变化也应反向提醒项目经理确认。

- 选择项目分类、金额分层和流程模板。
- 手动配置项目流程节点。
- 从商机或售前方案创建项目。
- 发起售前到实施交接。
- 登记合同评审结论和风险。
- 维护采购清单和到货计划。
- 确认设备到货并登记设备清单、序列号、授权和签收材料。
- 维护需求调研记录和需求确认。
- 确认方案功能范围和验收口径。
- 记录设备安装调试、联调结果和现场确认。
- 拆分任务。
- 更新任务状态。
- 上传会议录音并生成会议纪要。
- 确认会议纪要并同步项目干系人。
- 将纪要中的待办、风险、需求和客户反馈转为任务、风险、商机或互动记录。
- 将项目干系人同步到客户关键人关系链、权力地图和协作视图。
- 提交节点确认。
- 上传或关联节点输出产物。
- 发起变更申请并记录影响评估。
- 提交测试报告、客户培训记录、初验报告、验收单和运维移交单。
- 记录项目风险。
- 上传交付物。
- 发起项目复盘。
- 从项目复盘生成知识条目。

### 8.4 客户详情页

模块：

- 培训记录。

## 10. 规则体系

### 10.1 字段拆分

必须拆分两个字段：

- `signal_type`：商机/风险类型。
- `priority`：处理优先级。

不要把“类型”和“优先级”绑定。安全风险、新需求、竞品风险都可能因为客户等级、资产重要性、到期时间和重复发生情况而变成不同优先级。

### 10.2 MVP 规则表

| 规则编号 | 信号类型 | 触发条件 | 默认优先级 | 推荐动作 |
|---|---|---|---|---|
| R001 | 续约 | 维保/授权/证书到期 <= 30 天，可信度为 contract/confirmed | P0 | 联系客户确认续约或替代方案 |
| R002 | 续约 | 维保/授权/证书到期 <= 60 天，可信度为 imported/manual | P1 | 先核实到期日，再安排沟通 |
| R003 | 扩容 | 文本包含 CPU 高、内存不足、磁盘满、并发高、响应慢 | P1 | 安排技术诊断，判断扩容/优化方案 |
| R004 | 性能优化 | 同一客户同一资产 30 天内出现 3 次性能问题 | P1 | 输出问题分析，建议拜访 |
| R005 | 安全加固 | 文本包含漏洞、弱口令、被攻击、病毒、备份失败、日志审计 | P1 | 安排安全检查或加固方案 |
| R006 | 新需求 | 文本包含能不能、希望、想要、有没有、是否支持、领导要求 | P2 | 客户经理确认场景与预算 |
| R007 | 服务升级 | 文本包含响应慢、服务不及时、驻场、巡检频率 | P2 | 判断服务包升级可能 |
| R008 | 竞品/流失 | 文本包含竞品、替换、别家、重新招标 | P2/P3 | 主管或客户经理确认真实意图 |

### 10.3 优先级计算

优先级由以下因素共同决定：

- 到期时间。
- 客户等级。
- 资产重要性。
- 是否重复发生。
- 是否客户主动表达需求。
- 是否存在明确预算或采购计划。
- 数据可信度。

建议公式：

```text
priority_score =
  到期紧迫度 * 30
+ 客户等级 * 20
+ 资产重要性 * 20
+ 重复发生 * 15
+ 主动需求 * 10
+ 数据可信度 * 5
```

映射：

| 分数 | 优先级 |
|---:|---|
| 80-100 | P0 |
| 60-79 | P1 |
| 40-59 | P2 |
| 0-39 | P3 |

## 11. 商机状态机与 SLA

### 11.1 标准状态

| 状态 | 说明 | 可执行动作 | SLA |
|---|---|---|---|
| 待分配 | 信号已生成但没有负责人 | 分配负责人、关闭、误报 | P0/P1 24h |
| 待跟进 | 已有负责人，尚未开始处理 | 开始跟进、转派、误报、暂缓 | P0/P1 48h |
| 跟进中 | 负责人已开始联系客户 | 新增跟进、确认商机、放弃、暂缓 | 7 天 |
| 已确认商机 | 客户有明确意向 | 创建变更/项目、继续跟进、关闭 | 14 天 |
| 已转化 | 已进入变更/立项/报价流程 | 查看关联对象 | 无 |
| 已放弃 | 客户无意向或暂不推进 | 归档、恢复 | 无 |
| 误报 | 规则识别错误 | 进入误报反馈 | 30 天观察 |
| 已失效 | 长期未处理或未执行 | 恢复、归档 | 无 |

### 11.2 必填字段

| 动作 | 必填字段 |
|---|---|
| 开始跟进 | 下次跟进时间 |
| 确认商机 | 意向强度、预计金额区间、预计成交时间、下一步动作 |
| 放弃 | 放弃原因 |
| 暂缓 | 暂缓原因、重新提醒时间 |
| 误报 | 误报类型、说明 |
| 转派 | 新负责人、转派原因 |

## 13. 客户风险标签

MVP 不做复杂健康度分数，先做轻量风险标签。

| 标签 | 触发规则 |
|---|---|
| 健康 | 近 30 天无高优先级信号，无到期风险 |
| 关注 | 有 P2 信号，或 60 天内有到期资产 |
| 预警 | 有 P0/P1 信号，或 30 天内到期，或重复故障 |
| 商机 | 已确认商机或存在明确客户需求 |

客户详情页展示：

- 当前标签。
- 触发原因。
- 最近 3 条相关记录。
- 推荐动作。

## 14. 通知策略

### 14.1 通知渠道

MVP 优先级：

1. 系统内通知。
2. 企业微信机器人。
3. 邮件。
4. 短信暂缓，仅用于真正紧急的 P0。

### 14.2 通知规则

| 事件 | 通知对象 | 渠道 | 频率 |
|---|---|---|---|
| P0 信号生成 | 负责人、主管 | 系统内 + 企业微信 | 立即 |
| P1 信号生成 | 负责人 | 系统内 + 企业微信 | 立即 |
| P2 信号生成 | 负责人 | 系统内 | 每日汇总 |
| 无负责人 P0/P1 | 主管 | 系统内 + 企业微信 | 立即 |
| 48h 未处理 | 负责人、主管 | 系统内 + 企业微信 | 一次 |
| 跟进到期 | 负责人 | 系统内 | 每日 |
| 到期资产 30 天 | 负责人 | 系统内 + 企业微信 | 每周汇总 |

### 14.3 防打扰原则

- 同一客户同一类型信号 7 天内合并提醒。
- P2/P3 不实时打扰，只做汇总。
- 已确认跟进的信号不重复推送，只按下次跟进时间提醒。

### 15.1 数据范围

| 角色 | 客户范围 | 商机范围 |
|---|---|---|
| 运维工程师 | 自己负责或参与服务的客户 | 只看自己提交记录生成的信号摘要 |
| 项目经理 | 自己负责项目关联客户 | 自己负责客户的信号 |
| 客户经理/CSM | 分配给自己的客户 | 自己负责客户的信号 |
| 主管 | 本部门客户 | 本部门全部信号 |
| 管理员 | 全部 | 全部 |

### 15.2 敏感字段

默认脱敏：

- 合同金额。
- 联系人手机号。
- 客户原话中的账号、密码、IP、密钥。
- 日志、截图附件中的敏感信息。

原文查看权限：

- 普通负责人：查看脱敏文本。
- 主管：可查看原文。
- 管理员：可查看原文与审计日志。

MVP 阶段不保存原始语音。如后续必须保存，默认 30 天后自动删除。

## 17. 指标体系

### 17.1 北极星指标

每月被复用的服务资产数量 + 被持续维护的重点客户关系资产数量。

服务资产包括：有效项目复盘、有效知识条目、有效客户方案、有效培训记录、已确认商机、关键人关系链、有效互动记录和拜访复盘。该指标比单纯“商机数”更适合 v1.5，因为产品目标已经从商机雷达扩展为“项目交付 + 运维知识 + 区域 CRM + 客户经营 + 培训方案”的服务经营平台。

### 17.2 护栏指标

| 指标 | 定义 | 目标 |
|---|---|---:|
| 项目任务更新率 | 试点项目中按期更新状态的任务 / 总任务 | >= 80% |
| 项目风险关闭率 | 已关闭风险问题 / 总风险问题 | >= 70% |
| 有效运维记录数 | 试点期可被业务理解和复盘的运维记录 | 每周 >= 20 条 |
| 知识沉淀率 | 可沉淀记录中已生成知识 / 可沉淀记录 | >= 30% |
| 知识复用次数 | 新记录或新项目中引用历史知识次数 | 持续上升 |
| 培训记录覆盖率 | 有培训/方案记录的试点客户 / 试点客户 | >= 80% |
| 重点客户关系链覆盖率 | 已维护关键人关系链的重点客户 / 重点客户 | >= 80% |
| 拜访计划达成率 | 已完成拜访计划 / 到期拜访计划 | >= 75% |
| 拜访后记录回填率 | 48h 内完成互动记录的拜访 / 已完成拜访 | >= 80% |
| 90 天未互动重点客户占比 | 90 天无互动重点客户 / 重点客户 | <= 20% |
| 客户数据质量平均分 | 试点客户数据质量总分平均值 | >= 75 分 |
| 关键资产覆盖率 | 有核心资产数据的活跃客户 / 总活跃客户 | >= 70% |
| 信号响应及时率 | P0/P1 在 48h 内处理 / 总 P0/P1 | >= 80% |
| 已确认商机跟进率 | 已确认后有跟进记录 / 总已确认 | >= 70% |
| P0/P1 误报率 | 标记误报 / 总 P0/P1 信号 | <= 15% |
| 人工补录有效商机数 | 人工发现但系统未识别的有效商机 | 持续下降 |

### 17.3 漏检率口径

验证期漏检率建议定义为：

```text
人工补录有效商机数 /
（系统识别有效商机数 + 人工补录有效商机数）
```

## 19. 风险与依赖

| 风险 | 描述 | 缓解措施 |
|---|---|---|
| 运维记录录入摩擦高 | 运维人员可能不愿额外填写信息 | 模板点选、20 秒完成、不做考核、每周复盘选项 |
| 负责人不处理信号 | 项目经理/客户经理已有工作压力 | 首页待办、48h 提醒、主管分配池 |
| 规则误报透支信任 | 关键词规则早期可能误报 | P0/P1 宽松但可解释，P2 汇总，保留误报反馈 |
| 资产到期日不准 | 到期数据错误会导致续约信号错报 | 可信度标签、试点前清洗关键资产 |
| 与 M16 依赖不清 | 完整工单系统不在 MVP 范围内 | MVP 内置快速运维记录，不依赖 M16 |
| 客户健康度过重 | 复杂分数会拖慢 MVP | 第一版只做风险标签 |
| CRM 作战地图范围膨胀 | 容易扩展到完整销售自动化、报价、回款、外部数据抓取 | MVP 只做客户分布、关键人、拜访互动、轻量健康和数据质量 |
| 关键人信息敏感 | 联系方式、立场、家庭背景等信息可能引发隐私与合规风险 | 默认脱敏、最小必要字段、权限控制、操作审计，不记录无业务必要的敏感背景 |
| 关系链维护不持续 | 客户经理可能只在上线初期录入，后续不更新 | 低质量提醒、拜访后回填、月度区域复盘，把维护动作嵌入日常流程 |
| 项目管理过重 | 如果直接做甘特图、资源池、成本预算，会拖慢上线 | MVP 只做项目台账、任务、里程碑、风险和交付物 |
| 知识库无人维护 | 知识条目可能变成低质量堆积 | 设置维护人、状态、标签和定期复核 |
| 培训模块偏 LMS 化 | 如果做课程体系、考试、证书，会偏离客户经营目标 | 第一版只记录培训计划、材料、问题和后续动作 |
| 合同/电子签范围膨胀 | 参考上上签容易引入审批、签章、法律存证复杂能力 | MVP 只保留合同到期、归档链接和履约提醒字段 |

### SOP-01：快速运维记录补充流程

| Field | Value |
|---|---|
| Version | 1.0 |
| Last Updated | 2026-07-04 |
| Owner | 运维负责人 / 产品经理 |
| Frequency | 每次客户服务、巡检、故障处理或现场沟通后 |
| Estimated Time | 1-3 分钟 |

#### Purpose

确保运维工程师能用低摩擦方式记录客户现场事实，为商机雷达提供可识别的原始信号，同时避免让运维人员承担销售判断职责。

#### Scope

- Applies to：运维工程师、项目经理、客户经理/CSM。
- When：完成客户巡检、故障处理、咨询支持、变更支持、现场服务后。
- Not covered：完整工单处理、报价、合同、销售推进。

#### Prerequisites

- 已有客户档案。
- 如涉及具体系统，优先选择关联资产。
- 运维工程师有新增运维记录权限。

#### Procedure

**Step 1：进入快速记录入口**  
Who：运维工程师  
How：

1. 从首页、客户详情页或运维记录页点击“新增运维记录”。
2. 选择客户。
3. 如本次记录涉及具体系统，选择关联资产；如果不确定，可暂时留空。

**Step 2：选择记录类型与模板**  
Who：运维工程师  
How：

1. 选择记录类型：巡检、故障、咨询、变更、现场支持、其他。
2. 选择模板：性能问题、磁盘不足、证书到期、客户咨询、安全问题、新需求、投诉、竞品、其他。
3. 如果选择“其他”，必须填写不少于 5 个字的说明。

**Step 3：补充现场事实**  
Who：运维工程师  
How：

1. 填写问题描述，记录系统现象、客户反馈或现场观察。
2. 如有客户原话，填写在“客户反馈”字段。
3. 不需要判断是否为商机，也不要填写销售结论。

**Step 4：保存记录**  
Who：运维工程师  
How：

1. 点击保存。
2. 系统生成运维记录。
3. 系统自动进入规则识别流程。

**Decision Point**

- 如果记录命中规则：进入 SOP-02。
- 如果记录未命中规则：仅归档为普通运维记录。
- 如果客户或资产不存在：提交给主管或管理员补齐基础数据。

#### Checklist Summary

- [ ] 已选择客户。
- [ ] 已选择记录类型。
- [ ] 已选择模板。
- [ ] 已补充必要说明。
- [ ] 已保存记录。
- [ ] 如涉及资产，已关联资产或标记待补充。

#### Troubleshooting

| Problem | Likely Cause | Solution |
|---|---|---|
| 找不到客户 | 客户档案未创建或无权限 | 联系主管或管理员补齐客户档案 |
| 找不到资产 | 资产台账未维护 | 先保存记录，标记“资产待补充” |
| 不知道选哪个模板 | 场景不明确 | 选择“其他”，填写简短说明 |
| 记录包含账号/密码/IP | 涉及敏感信息 | 系统标记 sensitive_flag，按脱敏规则处理 |

#### Escalation

- Level 1：运维负责人。
- Level 2：产品/运营管理员。

### SOP-02：商机信号识别与生成流程

| Field | Value |
|---|---|
| Version | 1.0 |
| Last Updated | 2026-07-04 |
| Owner | 产品/运营管理员 |
| Frequency | 每次新增运维记录或资产到期任务触发时 |
| Estimated Time | 系统自动，人工复核 1-5 分钟 |

#### Purpose

将运维记录、资产到期和客户反馈转化为可解释的商机信号，确保每条信号都能说明触发原因、优先级和推荐动作。

#### Scope

- Applies to：规则引擎、产品/运营管理员、客户经理/CSM、主管。
- When：新增运维记录、资产到期扫描、规则配置变更后。
- Not covered：AI 自动判断、复杂预测模型、CRM 销售阶段管理。

#### Prerequisites

- 至少配置 8 条 MVP 规则。
- 客户档案存在。
- 运维记录或资产记录可被规则引擎读取。

#### Procedure

**Step 1：读取触发源**  
Who：系统  
How：

1. 读取新增运维记录或资产到期数据。
2. 提取客户、资产、记录类型、模板、描述、客户反馈、到期日、数据可信度。

**Step 2：执行规则匹配**  
Who：系统  
How：

1. 匹配到期类规则。
2. 匹配性能、扩容、安全、新需求、服务升级、竞品/流失关键词。
3. 判断是否存在重复发生：同一客户同一资产 30 天内是否出现多次同类问题。

**Step 3：计算优先级**  
Who：系统  
How：

1. 按到期紧迫度、客户等级、资产重要性、重复发生、主动需求、数据可信度计算 priority_score。
2. 将分数映射为 P0/P1/P2/P3。
3. 写入 trigger_reason 和 recommended_action。

**Step 4：生成商机信号**  
Who：系统  
How：

1. 如果命中规则，创建 opportunity_signal。
2. 如果已有同客户、同资产、同类型未关闭信号，则合并提醒，不重复生成。
3. 如果没有默认负责人，状态进入“待分配”。
4. 如果有默认负责人，状态进入“待跟进”。

**Decision Point**

- P0/P1：立即通知负责人和主管。
- P2：系统内通知或每日汇总。
- P3：MVP 阶段默认不主动推送，可进入低优先级列表。
- 重复信号：合并到既有信号，并追加 event_log。

#### Checklist Summary

- [ ] 已读取触发源。
- [ ] 已匹配规则。
- [ ] 已计算优先级。
- [ ] 已生成触发原因。
- [ ] 已生成推荐动作。
- [ ] 已写入商机池。
- [ ] 已按优先级触发通知。

#### Troubleshooting

| Problem | Likely Cause | Solution |
|---|---|---|
| 应该触发但没有触发 | 关键词缺失或资产数据缺失 | 管理员补充规则或资产字段 |
| 重复生成多个相同信号 | 去重规则缺失 | 按客户+资产+类型+未关闭状态合并 |
| 优先级明显过高 | 到期日或客户等级错误 | 修正基础数据并重新计算 |
| 误报过多 | 关键词过宽 | 进入 SOP-05 误报反馈流程 |

#### Escalation

- Level 1：产品/运营管理员。
- Level 2：产品经理。

### SOP-03：商机分配与跟进流程

| Field | Value |
|---|---|
| Version | 1.0 |
| Last Updated | 2026-07-04 |
| Owner | 客户成功主管 |
| Frequency | 每日 |
| Estimated Time | 单条 3-10 分钟 |

#### Purpose

确保商机信号生成后有人负责、有人响应、有人记录结果，避免信号停留在列表中无人处理。

#### Scope

- Applies to：主管、项目经理、客户经理/CSM。
- When：商机信号进入“待分配”“待跟进”“跟进中”状态时。
- Not covered：报价、合同签署、正式销售阶段管理。

#### Prerequisites

- 商机信号已生成。
- 客户存在负责人或可由主管分配负责人。
- 负责人有查看和处理商机权限。

#### Procedure

**Step 1：处理待分配信号**  
Who：主管  
How：

1. 打开首页“待分配信号”。
2. 查看客户、信号类型、优先级、触发原因。
3. 按负责人分配规则选择项目经理、客户经理或 CSM。
4. 点击“分配负责人”。

**Step 2：负责人开始跟进**  
Who：项目经理 / 客户经理 / CSM  
How：

1. 打开“商机池”或首页待办。
2. 查看触发原因、来源运维记录和推荐动作。
3. 点击“开始跟进”。
4. 填写下次跟进时间。

**Step 3：记录跟进结果**  
Who：负责人  
How：

1. 与客户沟通后，新增跟进记录。
2. 填写沟通内容、客户态度、下一步动作。
3. 根据结果选择：确认商机、放弃、暂缓、误报。

**Step 4：确认商机**  
Who：负责人  
How：

1. 点击“确认商机”。
2. 填写意向强度、预计金额区间、预计成交时间、下一步动作。
3. 如预计金额 >= 10 万，同步客户经理/销售负责人。
4. 后续进入报价、变更、立项或客户拜访。

**Decision Point**

- 如果 24h 内 P0/P1 无负责人：升级给主管。
- 如果 48h 内 P0/P1 未开始跟进：提醒负责人并抄送主管。
- 如果跟进中 7 天无新进展：提醒负责人更新状态。
- 如果已确认商机 14 天无真实业务动作：标记为待复核。

#### Checklist Summary

- [ ] 待分配信号已分配负责人。
- [ ] 负责人已查看触发原因。
- [ ] 已点击开始跟进。
- [ ] 已填写下次跟进时间。
- [ ] 已记录跟进结果。
- [ ] 已选择最终处理动作。

#### Troubleshooting

| Problem | Likely Cause | Solution |
|---|---|---|
| 无法判断分配给谁 | 客户归属不清 | 主管按客户阶段临时指定负责人 |
| 负责人长期未处理 | 工作负载过高或通知未读 | 主管转派或调整优先级 |
| 客户无明确意向 | 信号有价值但时机不成熟 | 标记暂缓并设置重新提醒时间 |
| 客户明确无需求 | 商机不成立 | 标记放弃并填写原因 |

#### Escalation

- Level 1：客户成功主管。
- Level 2：部门负责人。

### SOP-04：关键资产与到期数据维护流程

| Field | Value |
|---|---|
| Version | 1.0 |
| Last Updated | 2026-07-04 |
| Owner | 运维负责人 / 数据管理员 |
| Frequency | 初始导入、每月验证、客户服务后按需更新 |
| Estimated Time | 初始导入 30-60 分钟；单条维护 2-5 分钟 |

#### Purpose

保证维保、授权、证书等到期类数据准确，避免续约类 P0/P1 信号因数据错误而误报或漏报。

#### Scope

- Applies to：运维工程师、项目经理、管理员。
- When：新客户导入、服务合同签署、资产变化、到期日前验证、运维现场发现数据错误。
- Not covered：合同审批、采购流程、资产采购入库。

#### Prerequisites

- 客户档案已存在。
- 有关键系统、维保、授权或证书信息来源。
- 管理员或运维工程师具备资产维护权限。

#### Procedure

**Step 1：初始录入关键资产**  
Who：运维工程师 / 管理员  
How：

1. 新增客户关键系统资产。
2. 填写系统名称、类型、部署方式、维保到期日、证书到期日、状态。
3. 设置数据可信度：imported、manual、contract、confirmed。

**Step 2：每月验证到期数据**  
Who：运维工程师  
How：

1. 系统生成未来 90 天到期资产清单。
2. 运维工程师逐条确认或修改到期日。
3. 已确认数据标记为 confirmed。

**Step 3：现场发现数据错误时修正**  
Who：运维工程师 / 项目经理  
How：

1. 在客户详情或资产详情中点击“提交修正”。
2. 填写新日期和修正原因。
3. 项目经理或主管确认后生效。

**Decision Point**

- 可信度为 contract/confirmed 且 30 天内到期：触发 P0。
- 可信度为 imported/manual 且 60 天内到期：触发 P1，推荐先核实。
- 客户无关键资产数据：不生成 P0，进入“资产待补充”提醒。

#### Checklist Summary

- [ ] 客户关键资产已录入。
- [ ] 到期日已填写。
- [ ] 数据可信度已标记。
- [ ] 每月验证清单已处理。
- [ ] 修正记录有原因和确认人。

#### Troubleshooting

| Problem | Likely Cause | Solution |
|---|---|---|
| 客户无资产数据 | 历史数据未整理 | 标记资产待补充，先不生成 P0 |
| 到期日冲突 | 合同、证书、现场信息不一致 | 以合同或主管确认为准 |
| 已续约仍提醒 | 系统未更新到期日 | 提交修正并关闭旧信号 |
| 同一系统多个到期日 | 维保、授权、证书并存 | 分别记录，取最近日期触发提醒 |

#### Escalation

- Level 1：运维负责人。
- Level 2：数据管理员 / 项目经理。

### SOP-05：误报反馈与规则优化流程

| Field | Value |
|---|---|
| Version | 1.0 |
| Last Updated | 2026-07-04 |
| Owner | 产品/运营管理员 |
| Frequency | 按需标记，每周复盘 |
| Estimated Time | 单条 1-3 分钟；每周复盘 30 分钟 |

#### Purpose

让业务人员能快速反馈错误信号，让产品/运营管理员持续优化规则，避免误报透支用户信任。

#### Scope

- Applies to：项目经理、客户经理/CSM、主管、产品/运营管理员。
- When：负责人判断信号不应出现，或触发类型/优先级明显错误。
- Not covered：客户暂时无需求但规则识别合理的“已放弃”场景。

#### Prerequisites

- 商机信号已生成。
- 处理人有标记误报权限。
- 系统记录触发规则、原文和来源记录。

#### Procedure

**Step 1：标记误报**  
Who：负责人  
How：

1. 在商机详情中点击“误报”。
2. 选择误报类型：规则错误、时机不对、不相关、基础数据错误。
3. 填写简短说明。

**Step 2：系统记录误报证据**  
Who：系统  
How：

1. 保存 trigger_rule、source_text、source_record_id、marked_by、marked_at。
2. 信号状态变更为“误报”。
3. 写入 signal_event_log。

**Step 3：每周复盘误报**  
Who：产品/运营管理员  
How：

1. 导出本周误报列表。
2. 按规则编号和误报类型聚合。
3. 判断是否需要调整关键词、阈值、优先级或数据校验。

**Step 4：更新规则配置**  
Who：产品/运营管理员  
How：

1. 修改 rule_config。
2. 记录修改原因。
3. 对最近 30 天数据做简单回看，确认不会明显增加漏检。

**Decision Point**

- 如果是“基础数据错误”：转入 SOP-04。
- 如果是“客户暂时无预算”：不算误报，应标记“放弃”或“暂缓”。
- 如果同一规则一周内误报集中出现：优先调整规则。

#### Checklist Summary

- [ ] 已选择误报类型。
- [ ] 已填写说明。
- [ ] 已记录触发规则和来源文本。
- [ ] 已进入每周复盘。
- [ ] 规则调整已记录原因。

#### Troubleshooting

| Problem | Likely Cause | Solution |
|---|---|---|
| 用户把无需求标成误报 | 不理解误报定义 | 在界面提示“无需求请选择放弃/暂缓” |
| 误报集中在某类关键词 | 关键词过宽 | 收紧关键词或增加排除词 |
| 调整规则后漏检增加 | 收紧过度 | 回滚规则并补充分层优先级 |
| 无法追溯来源文本 | 日志缺失 | 补齐 source_record_id 和 trigger_rule |

#### Escalation

- Level 1：产品/运营管理员。
- Level 2：产品经理 / 客户成功主管。

### SOP-06：项目创建、分类与流程节点维护流程

| Field | Value |
|---|---|
| Owner | 项目经理 |
| Participants | 项目成员、实施、运维、客户经理、客户侧干系人 |
| Trigger | 新项目立项、项目分类变化、流程节点调整、任务拆分、客户交流汇报、项目会议或项目风险出现 |
| Frequency | 每个项目启动时创建；每次客户交流/项目会议后更新；每周至少复盘一次 |

#### Purpose

确保项目从前期调研、方案沟通、合同评审、售前交接、采购、需求调研、设备到货、方案功能确认、设备安装调试、沟通交付、变更、测试、客户培训、初验、验收、运维移交到复盘都有统一记录，并能按项目金额、项目类型和交付流程进行管理。每次客户交流、项目例会和阶段汇报都应沉淀会议纪要，避免项目过程只存在于群消息、个人表格、个人录音和零散文档中。

#### Procedure

1. 项目经理创建项目，填写客户、项目金额、金额分层、项目类型、项目阶段、计划周期和项目成员。
2. 项目经理选择默认全流程模板，或按实际项目手动配置流程节点。默认节点包括前期调研、方案沟通、合同评审、售前交接、项目采购、需求调研、设备到货确认、方案功能确认、设备安装调试、项目沟通交付、变更管理、联调测试、客户培训、初验、终验、运维移交。
3. 项目经理为每个节点设置负责人、计划时间、节点状态、是否需要确认、必须输出的产物类型。
4. 项目经理或节点负责人拆分任务，明确负责人、截止时间和交付物要求。
5. 项目成员按任务和节点更新进度，并上传方案、计划、纪要、配置清单、测试报告、验收材料、培训材料等交付物。
6. 采购或实施负责人在设备到货时登记设备型号、数量、序列号、授权许可、签收状态、到货照片和异常说明。
7. 项目经理组织方案功能确认，记录功能范围、功能边界、验收口径、客户确认意见和关联方案版本。
8. 实施负责人记录设备安装调试过程，包括安装位置、网络连通、参数配置、联调结果、问题整改和现场确认。
9. 培训负责人或项目经理记录客户培训对象、培训内容、培训材料、签到、客户问题、反馈和后续动作。
10. 每次客户交流、项目例会、阶段汇报或验收沟通后，会议负责人上传会议录音或录入会议内容。
11. 系统生成会议转写和结构化会议纪要，提取会议结论、待办事项、风险问题、客户反馈、下一步动作和参会干系人。
12. 项目经理或会议负责人确认纪要后，将纪要同步给项目成员、项目干系人、客户经理、客户侧参会人和被 @ 的协作成员。
13. 纪要中的待办事项可转为项目任务，风险问题可转为项目风险，客户反馈或新需求可转为商机信号或客户互动记录。
14. 纪要中的客户侧干系人应与客户管理中的关键人关系链同步；若不存在对应关键人，进入客户关系“待确认关键人”列表。
15. 客户管理中更新的关键人、权力地图或客户树变化，应提醒关联项目经理确认是否同步到项目干系人。
16. 节点进入“待确认”后，由确认人填写确认意见；需要确认的节点必须完成确认后才能进入“已确认”。
17. 项目出现风险、延期、客户阻塞或需求变化时，登记项目风险/问题。
18. 项目进入验收或关闭时，检查任务、流程节点、确认记录、会议纪要和交付物是否完整。
19. 项目关闭后发起复盘，沉淀问题、经验、方案模板和可复用知识。

#### Checklist Summary

- [ ] 项目已关联客户。
- [ ] 项目已填写金额、金额分层和项目类型。
- [ ] 已设置项目经理和项目成员。
- [ ] 已配置项目流程节点。
- [ ] 全流程节点已覆盖前期调研、方案沟通、合同评审、售前交接、采购、需求、设备到货、方案功能确认、安装调试、沟通交付、变更、测试、培训、初验、验收、运维移交。
- [ ] 每个节点已设置负责人、状态和计划时间。
- [ ] 需要确认的节点已记录确认人、确认时间和确认意见。
- [ ] 设备到货确认、方案功能确认、设备安装调试和客户培训记录已完成或有明确跳过原因。
- [ ] 关键交付物已上传或关联到节点。
- [ ] 每次关键客户交流或项目会议已生成并确认会议纪要。
- [ ] 会议纪要已同步给项目干系人和相关客户经理。
- [ ] 纪要中的待办、风险、客户反馈已转为任务、风险、商机或互动记录。
- [ ] 客户侧干系人与客户关键人关系链、权力地图和协作视图已同步或进入待确认。
- [ ] 风险问题已登记处理人和截止时间。
- [ ] 项目关闭前已完成复盘。

### SOP-07：运维知识沉淀流程

| Field | Value |
|---|---|
| Owner | 运维负责人 |
| Participants | 运维工程师、项目经理、产品/运营管理员 |
| Trigger | 运维记录关闭、巡检发现典型问题、项目复盘完成 |
| Frequency | 每周复盘一次知识候选 |

#### Purpose

把运维记录、巡检异常和项目复盘中的可复用经验沉淀为知识库，形成公司级服务资产。

#### Procedure

1. 系统根据记录类型、处理结果和复用价值标记“知识候选”。
2. 运维工程师从记录中点击“沉淀知识”。
3. 系统自动带入客户、资产、问题模板、处理过程和结果。
4. 维护人补充适用场景、处理步骤、注意事项、标签和附件。
5. 运维负责人审核后发布知识条目。
6. 后续新项目、新运维记录或培训问题可引用该知识。
7. 如果知识多次被反馈无效，状态变更为“需更新”。

#### Checklist Summary

- [ ] 来源记录完整。
- [ ] 处理步骤可复用。
- [ ] 已填写适用场景和标签。
- [ ] 已指定维护人。
- [ ] 已通过审核发布。

### SOP-08：客户培训与方案沉淀流程

| Field | Value |
|---|---|
| Owner | 培训负责人 |
| Participants | 项目经理、客户经理/CSM、运维工程师 |
| Trigger | 项目上线、客户新功能启用、客户提出培训需求、服务升级前 |
| Frequency | 每次客户培训后更新 |

#### Purpose

把客户培训从一次性会议转化为客户经营资产，沉淀培训材料、客户问题、方案模板和后续服务机会。

#### Procedure

1. 培训负责人创建培训计划，关联客户、项目、培训主题和讲师。
2. 选择或上传培训材料，可关联方案库模板。
3. 培训完成后记录参训人员、客户问题、培训反馈和后续动作。
4. 客户问题可整理为 FAQ 知识条目。
5. 如果培训中出现新需求、扩容、安全加固、服务升级等信号，生成商机信号或人工补录。
6. 培训记录沉淀到客户详情页，并进入客户简报素材。

#### Checklist Summary

- [ ] 培训计划已关联客户和项目。
- [ ] 培训材料已上传或关联方案库。
- [ ] 已记录参训人员。
- [ ] 已记录客户问题和后续动作。
- [ ] 可复用问题已沉淀为 FAQ。
- [ ] 可能商机已进入商机池或人工补录。

### SOP-09：区域客户作战地图维护流程

| Field | Value |
|---|---|
| Owner | 区域负责人 |
| Participants | 客户经理/CSM、主管、产品/运营管理员 |
| Trigger | 新客户导入、客户归属调整、月度区域复盘、重点客户状态变化 |
| Frequency | 每周检查异常；每月正式复盘 |

#### Purpose

保证区域作战地图中的地市、行业、客户等级、负责人、风险状态和数据质量持续准确，让客户分布和经营状态可视化、可复盘。

#### Procedure

1. 管理员或客户经理导入客户基础数据，至少填写客户名称、地市、行业、客户等级、负责人和服务状态。
2. 系统按地市、行业、等级和负责人生成客户地图及列表。
3. 系统识别缺失地市、负责人、客户等级、关键人或最近互动的客户，进入“待补充”列表。
4. 客户经理每周处理自己名下的待补充客户，补齐字段或提交无法补齐原因。
5. 区域负责人每月查看地市覆盖、重点客户、预警客户、有效商机、拜访达成和数据质量。
6. 对重点客户覆盖不足、长期未互动、低质量数据或无人负责客户，区域负责人指定处理人和完成时间。
7. 月度复盘结果导出，作为下月客户经营计划输入。

#### Checklist Summary

- [ ] 客户已标注地市、行业、等级和负责人。
- [ ] 重点客户已进入重点经营清单。
- [ ] 低数据质量客户已有补充计划。
- [ ] 90 天未互动重点客户已分配维护动作。
- [ ] 月度区域复盘已导出。

### SOP-10：客户拜访计划与互动回填流程

| Field | Value |
|---|---|
| Owner | 客户经理/CSM |
| Participants | 项目经理、运维工程师、培训负责人、主管 |
| Trigger | 重点客户维护、商机跟进、项目关键节点、培训前后、客户风险预警 |
| Frequency | 每次客户拜访或互动前后 |

#### Purpose

把客户拜访和日常互动从个人记忆转为系统记录，确保拜访有目的、互动有结果、下一步有提醒。

#### Procedure

1. 客户经理创建拜访计划，填写客户、互动对象、目的、计划时间、准备材料和预期结果。
2. 如拜访与商机、项目、培训、客户风险相关，必须关联对应记录。
3. 系统在计划时间前提醒负责人准备客户简报、方案材料或项目资料。
4. 互动完成后 48 小时内回填互动记录，填写互动类型、对象、关键内容、客户反馈、下一步动作和下次跟进时间。
5. 如果互动中出现续约、扩容、安全加固、服务升级、竞品替换等信息，创建商机信号或关联已有商机。
6. 系统更新客户最近互动时间、关系亲疏度、风险标签和客户详情页时间线。
7. 逾期未回填时提醒负责人，超过 72 小时升级给主管。

#### Checklist Summary

- [ ] 拜访计划已明确目的和对象。
- [ ] 已准备客户简报或方案材料。
- [ ] 互动后 48 小时内已回填记录。
- [ ] 下一步动作和下次跟进时间已填写。
- [ ] 新需求或风险已转为商机、任务或风险记录。

### SOP-11：权力地图、客户树与协作视图维护流程

| Field | Value |
|---|---|
| Owner | 客户经理/CSM |
| Participants | 区域负责人、主管、项目经理、运维负责人 |
| Trigger | 新增重点客户、项目启动、商机确认、客户组织变化、客户树变化、协作任务产生、客户交接 |
| Frequency | 重点客户每月至少检查一次；发生变化时即时更新 |

#### Purpose

沉淀客户内部决策链、跨客户关系和团队协作过程，降低人员离职、客户经理变更或项目交接造成的关系资产流失。

#### Procedure

1. 客户经理在重点客户详情页新增关键人，填写姓名、职务、部门和角色类型。
2. 按业务事实标注角色：决策人、影响人、使用人、信息人、反对者或其他。
3. 记录权力等级、影响力、亲疏度、立场、决策角色、关键诉求和最近互动，不记录无业务必要的隐私信息。
4. 在权力地图中维护关键人节点、上下级关系、影响路径、支持/反对立场和决策链路。
5. 在客户树中维护集团、上下级单位、关联单位、供应链、竞争关系、人事流动等跨客户关联。
6. 团队成员可在客户、关键人、关系线、客户树节点、拜访计划或商机上添加评论、标注、协作任务和 @ 成员。
7. 系统检查重点客户是否至少包含 1 个决策人和 1 个影响人或使用人。
8. 项目经理、运维工程师或培训负责人发现关键人或客户树变化时，提交关系链更新建议。
9. 客户经理确认后更新关键人、客户树或协作任务状态，并写入操作日志。
10. 客户交接时，接手人必须查看权力地图、客户树、协作任务、最近互动、商机和风险后确认接收。

#### Checklist Summary

- [ ] 重点客户已维护关键人。
- [ ] 关键人角色和权力等级已标注。
- [ ] 权力地图已维护关键人节点、影响路径和决策链路。
- [ ] 客户树已维护集团、上下级或其他跨客户关联。
- [ ] 协作评论、标注和任务已明确负责人和状态。
- [ ] 最近互动和关键诉求已记录。
- [ ] 关系链缺口已进入待补充提醒。
- [ ] 客户交接时权力地图、客户树和协作任务已确认。

## 23. 前端交互设计规范（来自 CRM PRD v1.2）

> **说明**：本章节基于 Demo 实现，定义前端各页面的交互标准、组件规范和操作反馈机制。所有 23 个 HTML 页面统一使用共享组件库 `ui-components.js` 提供 Toast、Modal、ConfirmDialog、FormValidator、Loading、DataTable 等全局交互能力。

### 23.1 交互设计原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **即时反馈** | 用户操作后 300ms 内给出反馈 | 按钮点击 scale 动画、Toast 通知 |
| **防错设计** | 危险操作需二次确认 | 删除操作弹出 ConfirmDialog |
| **渐进展示** | 复杂信息分层展示，避免信息过载 | 列表 → 详情 Modal → 编辑表单 |
| **统一语言** | 相同操作使用相同的交互模式 | 所有"新建"都用 Modal 表单 |
| **容错恢复** | 误操作可撤销或恢复 | 协作项勾选可取消 |

### 23.2 全局交互组件

系统基于共享组件库 `ui-components.js` 提供以下全局交互能力：

#### 23.2.1 Toast 通知系统

| 类型 | 用途 | 自动消失时间 |
|------|------|------------|
| success | 操作成功反馈 | 3秒 |
| error | 操作失败/验证错误 | 5秒 |
| warning | 警告提示 | 4秒 |
| info | 一般信息提示 | 3秒 |

**使用场景：**
- 新建/编辑/删除成功 → `Toast.success()`
- 表单验证失败 → `Toast.error()`
- 筛选条件切换 → `Toast.info()`
- 数据导出提交 → `Toast.success('导出任务已提交')`

#### 23.2.2 Modal 模态框系统

| 类型 | 宽度 | 用途 |
|------|------|------|
| 表单型 | 420-480px | 新建/编辑表单 |
| 详情型 | 480-520px | 数据详情展示 |
| 确认型 | 380px | 删除/重要操作确认 |
| 预览型 | 600px | 数据预览/报表预览 |

**Modal 交互规范：**
- 点击遮罩层关闭
- 按 ESC 键关闭
- 关闭按钮始终可见（确认型除外）
- 表单型 Modal 关闭前如有未保存数据需提示

#### 23.2.3 ConfirmDialog 确认对话框

**使用场景：**
- 删除客户/项目/任务
- 批量操作确认
- 状态变更确认（如项目归档）

**按钮规范：**
- 左侧：取消按钮（灰色边框）
- 右侧：确认按钮（红色/橙色，根据危险程度）

#### 23.2.4 FormValidator 表单验证

**验证规则：**
| 规则 | 说明 | 示例 |
|------|------|------|
| required | 必填 | 客户名称、手机号 |
| email | 邮箱格式 | 联系人邮箱 |
| phone | 手机号格式 | 联系方式 |
| min | 最小长度 | 密码 ≥ 6 位 |
| max | 最大长度 | 备注 ≤ 500 字 |
| number | 数字格式 | 金额、进度 |

**验证时机：**
- 提交时统一验证
- 验证失败时字段边框变红，下方显示错误提示
- 错误提示 11px 红色文字

#### 23.2.5 Loading 加载状态

**使用场景：**
- 数据导出 → `Loading.show('正在导出...')` → 完成后 `Loading.hide()`
- 数据同步 → `Loading.show('正在同步...')`
- 大数据量筛选 → `Loading.show('正在筛选...')`

### 23.3 页面级交互规范

#### 23.3.1 列表页交互标准

所有包含列表的页面统一遵循以下交互模式：

| 交互 | 操作 | 反馈 |
|------|------|------|
| 搜索 | 输入关键词 | 防抖 300ms 后实时筛选 |
| 筛选 | 点击筛选按钮/下拉选择 | 高亮当前筛选 + Toast 提示 |
| 排序 | 点击表头 | 升序/降序切换，当前列标记箭头 |
| 分页 | 点击页码 | 切换页码高亮 + Toast 提示 |
| 行详情 | 点击"详情" | Modal 打开详情面板 |
| 行编辑 | 点击"编辑" | Modal 打开编辑表单（预填数据） |
| 行删除 | 点击"删除" | ConfirmDialog 确认 → 行淡出动画 → Toast |
| 新建 | 点击"新建" | Modal 打开新建表单 |

#### 23.3.2 看板页交互标准

| 交互 | 操作 | 反馈 |
|------|------|------|
| 卡片点击 | 点击任务卡片 | Modal 打开任务详情 |
| 卡片拖拽 | 按住卡片拖动 | 视觉反馈（提升/透明度变化） |
| 筛选 | 点击优先级/标签 | 看板只显示匹配的卡片 |
| 新建 | 点击"新建任务" | Modal 打开任务表单 |
| 预警 | 截止日期临近 | 卡片边框红色高亮 |

#### 23.3.3 表单页交互标准

| 交互 | 操作 | 反馈 |
|------|------|------|
| 提交 | 点击提交按钮 | 验证 → 通过则关闭 Modal + Toast.success |
| 取消 | 点击取消/关闭 | 直接关闭 Modal |
| 验证错误 | 字段不符合规则 | 边框变红 + 错误提示 + Toast.error |
| 字段联动 | 选择客户后 | 自动填充客户相关信息 |

### 23.4 动画与过渡规范

| 元素 | 动画 | 时长 | 说明 |
|------|------|------|------|
| Modal 打开 | 遮罩淡入 + 内容缩放 | 200ms | 从 scale(0.95) 到 scale(1) |
| Modal 关闭 | 遮罩淡出 + 内容缩放 | 200ms | 从 scale(1) 到 scale(0.95) |
| Toast 出现 | 右侧滑入 | 300ms | 从 translateX(100%) 到 translateX(0) |
| Toast 消失 | 右侧滑出 | 300ms | 从 translateX(0) 到 translateX(100%) |
| 行删除 | 透明度渐变 | 300ms | 从 opacity(1) 到 opacity(0) |
| 按钮点击 | 缩放反馈 | 100ms | 从 scale(1) 到 scale(0.97) 再恢复 |
| 进度条 | 宽度动画 | 500ms | 从 0% 动画到实际进度 |
| 筛选切换 | 背景色过渡 | 200ms | 按钮背景色渐变 |
| 下拉菜单 | 淡入 | 150ms | 从 opacity(0) 到 opacity(1) |

## 24. API 接口规范（来自 CRM PRD v1.2）

> **说明**：本章节定义前端页面与后端交互的 RESTful API 接口规范，供开发团队参考实现。

### 24.1 接口通用规范

**基础路径：** `/api/v1`

**请求格式：**
- Content-Type: `application/json`
- 认证方式：Bearer Token（JWT）

**响应格式：**
```json
{
  "code": 200,
  "message": "success",
  "data": { ... }
}
```

**错误码：**
| 错误码 | 说明 | 前端处理 |
|--------|------|---------|
| 200 | 成功 | 正常处理 |
| 400 | 请求参数错误 | Toast.error('请检查输入') |
| 401 | 未认证 | 跳转登录页 |
| 403 | 无权限 | Toast.error('无操作权限') |
| 404 | 资源不存在 | Toast.error('数据不存在') |
| 409 | 数据冲突 | Toast.error('数据已存在') |
| 500 | 服务器错误 | Toast.error('系统繁忙，请稍后') |
