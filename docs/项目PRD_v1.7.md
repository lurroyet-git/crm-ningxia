## 1. 文档目的

本文档用于定义“项目管理工具 + 商机雷达”融合方案从单点 M23 MVP 向综合业务平台演进后的产品方案。v1.7 在 v1.5 基础上对齐实际前端UI实现，修正功能描述与界面不一致之处，补充已实现功能（通知系统、数据导出、团队设置、资产台账、方案素材点赞下载等），并标注信号驱动商机、AI会议纪要、数据质量评分等需要后续迭代的差异。

v1.7 保留核心业务逻辑，同时对齐实际前端UI实现。不再只描述一个轻量商机雷达，而是定义一套面向信息化服务公司的业务工作台。根据当前 Demo，产品主导航收敛为六个一级模块：作战台、项目交付、运维管理、商机营销、客户资产、知识分享。它仍然保留第一期 MVP 的收敛原则：先让项目、运维、客户、关系和商机数据在一个系统里形成闭环，再逐步扩展完整合同、报价、审批、AI 和自动化变动抓取能力。

原 M23 MVP 的最小闭环仍然成立：

`客户档案 -> 关键资产 -> 运维记录 -> 规则识别 -> 商机信号 -> 负责人跟进 -> 结果回填`

v1.7 扩展后的主闭环为：

`项目立项 -> 任务交付 -> 运维巡检 -> 记录沉淀 -> 知识复用 -> 商机识别 -> 区域作战地图 -> 客户拜访/关系维护 -> 培训/方案交付 -> 续约/扩容/服务升级`

本文档将作为研发排期、测试验收、试点上线和业务评审的依据。

## 2. 背景与问题

信息化公司在项目交付、运维服务、客户经营之间通常存在明显断层。运维人员最接近客户现场，最早知道系统性能、资产到期、安全隐患、客户抱怨和新需求，但这些信息如果只停留在工单、微信群、个人笔记或口头沟通里，就很难转化为可跟进的续约、扩容、安全加固、系统改造和服务升级机会。

原 v1.2 方案已完整描述平台远景，但范围过大，第一期容易被完整项目管理、工单、变更、AI、大屏、项目集等能力拖住。v1.3 将 M23 独立收敛为轻量 MVP，先验证“运维记录能否稳定产生真实商机”。

## 3. 产品定位

### 3.1 一句话定位

面向信息化服务公司的**项目交付、运维服务、客户经营和培训交付一体化平台**。

核心价值：将项目过程、运维记录、巡检事实、知识经验、客户培训和商机线索沉淀到同一个业务闭环，帮助团队把服务交付转化为可复用知识和可跟进的增长机会。

### 3.2 六大工作模块

| 一级模块 | 核心问题 | 产品能力 | 业务结果 |
|---|---|---|---|
| 作战台 | 团队每天不知道优先处理什么 | 今日概览、待处理、经营报表、数据导出、团队设置 | 一屏聚合项目、运维、客户、商机和提醒 |
| 项目交付 | 项目前期调研、售前方案、合同评审、实施交接、采购、需求、变更、测试、验收、运维移交分散在表格和群消息中 | 项目看板、任务排期、交付简报、全流程节点、会议纪要、风险问题、交付物 | 项目从售前到运维可视、责任清晰、交付经验可复用 |
| 运维管理 | 运维经验只留在个人、工单或聊天记录里 | 运维记录、巡检计划、巡检记录、资产台账、知识沉淀、规则配置 | 形成公司级运维知识资产，并从现场事实识别机会 |
| 商机营销 | 客户现场信号无法转成续约、扩容、服务升级机会 | 商机池、营销跟进、拜访计划、来源证据链和下次动作 | 提升续约、扩容、安全加固和服务升级转化 |
| 客户资产 | 客户资料、区域分布、关键人关系和互动记录不可交接 | 客户档案、区域地图、权力地图、客户树、关系网络、可视化协作、数据质量和互动历史 | 客户关系从个人经验转为组织资产 |
| 知识分享 | 培训材料、客户方案、交付文档无法系统复用 | 培训计划、方案素材、FAQ、知识引用和客户简报素材 | 降低培训交付成本，提高客户使用深度 |

### 3.3 参考产品启发

本产品不直接复制某一类工具，而是组合学习以下产品模式：

| 参考方向 | 可借鉴点 | 本产品取舍 |
|---|---|---|
| 金蝶类项目管理/企业协同工具 | 项目全生命周期、任务责任、交付物、风险预警、业财和合同衔接 | 先做项目台账、任务、里程碑、风险和交付物，不在 MVP 做复杂预算、成本、回款和 ERP 深集成 |
| 上上签类电子签约/合同流程工具 | 合同全生命周期、审批签署、存证、到期提醒、履约节点 | 先借鉴“合同/维保到期提醒、存证和流程留痕”，不在 MVP 做电子签章、法律存证和合同审批 |
| 飞书项目/飞书知识库、Notion、Confluence 类协同知识产品 | 项目协同、结构化文档、知识库、模板复用、跨团队协作 | 先做轻量知识条目、方案模板、项目复盘，不做复杂文档权限和多人编辑器 |
| 禅道、Tapd、Jira 类项目/任务工具 | 任务流转、缺陷问题、迭代看板、负责人和状态 | 保留任务和风险问题管理，不把产品做成研发缺陷管理工具 |
| 客户成功/CRM 类工具 | 客户档案、跟进记录、健康标签、续约和扩容机会 | 只做服务型客户经营，不替代完整 CRM 销售漏斗 |
| 纷享销客、Salesforce、Pipedrive、Monday CRM、Zoho CRM | 权力地图、客户树、销售管道、可视化协作、数据质量和健康度模型 | 先做区域客户地图、关键人关系链、拜访互动和轻量健康度，不在 MVP 做完整销售自动化 |

### 3.4 产品边界

v1.7 的产品定位是“服务交付与客户经营一体化工作台”。M23 商机雷达是其中的客户经营子模块，区域 CRM 作战地图是客户经营的组织化载体。

它不做：

- 不替代完整 ERP、财务、采购和成本核算系统。
- 不替代完整 CRM 销售流程和报价回款流程。
- 不替代复杂 ITSM 工单系统。
- 不承担合同回款、报价审批和电子签章法律能力。
- 不要求运维人员承担销售职责。
- 不在 MVP 阶段做 AI 销售顾问、AI 培训讲师、复杂知识图谱或外部人事变动自动抓取。

它只做：

- 汇集项目、客户、资产、运维记录、巡检计划、巡检记录和培训记录。
- 用轻量任务和里程碑管理项目交付过程。
- 将运维记录沉淀为可检索、可复用的知识和方案。
- 用规则识别高频巡检商机信号，并推动跟进。
- 用区域 CRM 作战地图呈现客户分布、关系链、拜访计划和重点客户状态。
- 记录客户培训、方案交付和客户能力建设过程。
- 为后续 CRM、工单、合同、电子签、AI、客户健康度和业财集成打基础。

## 4. MVP 目标与非目标

### 4.1 业务目标

在 8-10 周内上线一个可试点的 Web MVP，支持 10-20 个真实客户、3-5 个进行中项目、2-3 名运维工程师、1-2 名项目经理、1-2 名客户经理/CSM、1 名培训负责人、1 名区域负责人和 1 名主管完成试运行。

第一期目标不是做“大而全项目管理平台”，而是验证四个闭环是否能跑通：

1. 项目任务能否沉淀为可复盘的交付过程。
2. 运维记录能否稳定沉淀为知识和方案。
3. 巡检记录能否稳定产生真实商机。
4. 客户培训能否关联客户、项目、方案和后续服务机会。
5. 客户关系、关键人和拜访互动能否沉淀为可交接的组织资产。

### 4.2 验证假设

| 假设 | 验证方式 | 通过标准 |
|---|---|---|
| 运维现场信息能稳定产生商机线索 | 持续录入运维记录并生成信号 | 每月不少于 5 条可跟进商机 |
| 运维人员愿意低摩擦补充记录 | 快速记录入口、模板点选、非强制长文本 | 补充记录完成率 > 50% |
| 项目经理/客户经理愿意处理信号 | 商机池、提醒、分配、状态流转 | P0/P1 信号 48h 内响应率 > 80% |
| 规则识别不会明显透支信任 | 误报反馈、人工确认 | P0/P1 误报率 < 15% |
| 到期资产能带来明确续约动作 | 关键资产录入、到期提醒 | 关键资产覆盖率 > 70% |
| 项目过程能被项目经理持续维护 | 项目台账、任务、里程碑、风险记录 | 试点项目任务更新率 > 80% |
| 运维知识能被复用 | 运维记录转知识、知识检索、方案引用 | 每月至少沉淀 10 条有效知识 |
| 客户培训能促进后续服务 | 培训计划、培训记录、方案推荐 | 每个试点客户至少 1 条培训/方案记录 |
| 区域客户关系能被组织化管理 | 重点客户关系链、拜访记录、数据质量评分 | 重点客户关系链覆盖率 >= 80% |
| 客户拜访动作能被复盘 | 拜访计划、互动记录、下次动作 | 拜访后 48h 记录回填率 >= 80% |

### 4.3 非目标

MVP 阶段不做以下能力：

- 完整 CRM 销售流程。
- 外部政府网站、脉脉、LinkedIn 的自动变动抓取。
- 完整工单系统。
- 合同回款、报价、审批。
- 电子签章、合同法律存证。
- 复杂项目成本、预算、回款和业财一体化。
- 多项目集资源池和复杂排期算法。
- 在线考试系统和完整 LMS 学习平台。
- AI 销售顾问。
- 客户健康度复杂评分。
- 复杂图数据库关系推理和自动客户树发现。
- 大屏。
- 移动 App。
- 项目集协同。
- 规则版本回滚。
- 复杂权限审批流。
- M12 变更自动创建。
- M16 完整工单关闭弹窗。

## 5. 用户角色与职责

### 5.1 角色定义

| 角色 | 核心职责 | MVP 中的动作 |
|---|---|---|
| 运维工程师 | 记录现场事实，不承担销售职责 | 新增运维记录、选择问题模板、补充客户反馈 |
| 项目经理 | 判断交付期客户风险与机会 | 查看信号、确认/放弃、创建跟进 |
| 客户经理/CSM | 负责客户沟通与商机推进 | 接收分配、记录跟进、填写意向 |
| 区域负责人 | 负责区域客户覆盖、重点客户推进和跨团队协同 | 查看区域作战地图、识别空白客户、复盘拜访和机会 |
| 主管 | 分配无负责人信号、处理超时 | 查看待分配、查看超时、调整负责人 |
| 产品/运营管理员 | 维护规则和基础配置 | 配置关键词、查看误报、维护字典 |

### 5.2 负责人分配规则

| 场景 | 默认负责人 | 兜底 |
|---|---|---|
| 项目执行期客户 | 项目经理 | 部门主管 |
| 运维服务期客户 | 客户经理/CSM | 部门主管 |
| 纯维保客户 | 服务合同负责人 | 客户经理 |
| 无负责人客户 | 进入主管分配池 | 24h 未分配升级 |
| 预计金额 >= 10 万 | 同步客户经理/销售负责人 | 主管确认 |

### 6.3 暂缓

- P3 主管待审池。
- 规则版本管理与回滚。
- 复杂客户健康度四维度加权模型。

## 7. 核心业务流程

### 7.1 主流程

1. 运维工程师在快速记录入口新增一条运维记录。
2. 系统保存客户、资产、问题模板、文本描述和客户反馈。
3. 规则引擎根据到期时间、关键词、重复故障等条件识别信号。
4. 系统生成商机信号，写入商机池。
5. 系统按负责人分配规则设置 owner；无负责人进入主管分配池。
6. 负责人在商机池中点击“开始跟进”，填写下次跟进时间。
7. 负责人根据沟通结果选择“确认商机 / 放弃 / 暂缓 / 误报”。
8. 已确认商机进入后续真实业务动作：报价、变更、立项或客户拜访。
9. 系统持续记录状态变化、跟进记录和通知日志。

### 7.2 运维记录原则

运维端只记录事实，不要求判断商机。界面文案应避免“销售”“商机转化”等表达，优先使用：

- 现场补充
- 客户反馈
- 风险线索
- 系统现象
- 后续建议

### 7.3 项目管理闭环

1. 项目经理创建项目，选择客户、项目阶段、项目成员和计划周期。
2. 项目经理拆分里程碑和任务，明确负责人、截止时间和交付物。
3. 成员按任务更新进度，上传方案、计划、会议纪要、验收材料等交付物。
4. 项目风险或问题出现时，项目经理登记风险、影响范围、处理人和截止时间。
5. 项目进入上线、试运行、验收等节点时，系统提醒负责人补齐交付物。
6. 项目结束后触发复盘流程，沉淀项目经验、问题清单、可复用方案和培训材料。
7. 复盘内容可进入知识库，并可关联到客户、行业、产品模块和后续项目。

### 7.4 运维知识沉淀闭环

1. 运维工程师新增运维记录或巡检记录。
2. 系统保存客户、资产、问题模板、处理过程、客户反馈和结果。
3. 记录关闭后，如果满足“可复用”条件，系统提醒沉淀知识。
4. 运维人员或主管将记录整理为知识条目：适用场景、处理步骤、注意事项、附件。
5. 知识条目通过标签、客户、行业、资产、问题模板进行分类。
6. 后续新增运维记录、巡检记录或培训问题时，系统可检索并推荐相关知识。
7. 如果知识被多次引用或反馈无效，进入知识维护提醒。

### 7.5 客户培训与方案闭环

1. 项目经理、客户经理或培训负责人为客户创建培训计划。
2. 培训计划关联客户、项目、产品模块、方案材料和讲师。
3. 培训后记录参训人员、客户问题、培训反馈和后续动作。
4. 客户问题可沉淀为 FAQ 或知识条目。
5. 如果培训中出现新需求、服务升级、扩容、安全加固等信号，系统生成商机信号或人工补录商机。
6. 培训记录沉淀到客户详情页，作为后续续约、复购、项目复盘和客户简报素材。

### 7.6 区域 CRM 作战地图闭环

1. 客户经理或管理员导入客户基础数据，补充地市、行业、等级、负责人和服务状态。
2. 系统按地市、行业、客户等级、负责人和风险标签生成区域作战地图。
3. 客户经理为重点客户维护关键人关系链，至少包含决策人、影响人、使用人或信息人。
4. 客户经理制定拜访或互动计划，明确拜访目的、对象、材料和下一步动作。
5. 拜访、电话、微信、会议、培训等互动完成后，48 小时内回填互动记录。
6. 系统根据最近互动、关系链完整度、项目/商机风险和数据质量更新客户状态。
7. 区域负责人按月复盘地市覆盖、重点客户健康度、拜访达成、商机推进和数据质量。

作为运维工程师，我希望在处理完客户问题后，用 10-20 秒记录现场信息，这样公司能发现潜在风险和商机，而我不用写复杂销售材料。

验收标准：

- 可以选择客户和资产。
- 可以选择模板：性能问题、磁盘不足、证书到期、客户咨询、安全问题、新需求、其他。
- 选择“其他”时必须填写 5 个字以上说明。
- 保存后生成运维记录。
- 若命中规则，系统自动生成商机信号。

### US-02 客户经理处理商机信号

作为客户经理，我希望看到系统识别出的商机信号、触发原因和推荐动作，这样我能判断是否要跟进客户。

验收标准：

- 商机池显示待跟进信号。
- 可查看触发原因和来源运维记录。
- 可点击“开始跟进”，状态变为“跟进中”。
- 系统要求填写下次跟进时间。
- 超过下次跟进时间未处理时进入提醒。

### US-03 主管分配无负责人信号

作为主管，我希望看到没有负责人的高优先级信号，并快速分配给客户经理或项目经理。

验收标准：

- 首页显示“待分配信号”。
- 可批量选择负责人。
- 分配后负责人收到提醒。
- 24 小时未分配的 P0/P1 信号进入升级提醒。

### US-04 管理员维护识别规则

作为产品/运营管理员，我希望维护关键词和优先级规则，这样系统能持续适配业务场景。

验收标准：

- 可新增/停用关键词。
- 可配置关键词对应的信号类型和默认优先级。
- 可查看规则命中次数和误报次数。
- 可导出误报列表。

### US-05 项目经理维护项目交付过程

作为项目经理，我希望在一个页面按项目金额、项目类型和全生命周期流程管理项目，并能覆盖前期调研、方案沟通、合同评审、售前到实施交接、项目采购、需求调研、设备到货、方案功能确认、设备安装调试、项目沟通交付、变更、测试、客户培训、初验、验收和运维移交，这样我能及时推动项目交付，并在项目结束后形成可复用经验。

验收标准：

- 可创建项目并关联客户。
- 可填写项目金额、金额分层、项目类型和交付复杂度。
- 可按软件、系统集成、数字化转型、运维服务、培训/专项服务等分类筛选项目。
- 可手动新增、删除、排序和改名项目流程节点。
- 每个流程节点可设置负责人、计划时间、节点状态、是否需要确认和必需输出产物。
- 默认流程模板应包含前期调研、方案沟通、合同评审、售前交接、项目采购、需求调研、设备到货确认、方案功能确认、设备安装调试、项目沟通交付、变更管理、联调测试、客户培训、初验、终验、运维移交。
- 需要确认的节点可记录确认人、确认时间和确认意见。
- 设备到货节点可登记设备型号、数量、序列号、授权、签收状态和到货照片/附件。
- 方案功能确认节点可记录功能范围、功能边界、验收口径和客户确认意见。
- 设备安装调试节点可记录安装位置、网络连通、参数配置、调试结果和现场确认。
- 客户培训节点可记录培训对象、培训材料、签到、问题反馈和后续动作。
- 可创建任务并分配负责人和截止时间。
- 可登记项目风险或问题。
- 可上传或关联交付物，并绑定到具体流程节点。
- 可上传会议录音并生成会议转写和结构化会议纪要。
- 会议纪要可提取会议结论、待办事项、风险问题、客户反馈和参会干系人。
- 会议纪要确认后可同步给项目成员、项目干系人、客户经理和被 @ 成员。
- 会议中识别出的客户侧干系人可同步到客户关键人关系链、权力地图和客户协作视图。
- 客户管理中更新的关键人关系变化可提醒关联项目经理确认是否同步到项目干系人。
- 项目结束后可发起复盘并生成知识条目。

### US-06 运维人员沉淀知识

作为运维工程师，我希望把已解决的问题快速转成知识条目，这样下次遇到类似问题时可以直接复用处理步骤。

验收标准：

- 运维记录关闭后可点击“沉淀知识”。
- 系统自动带入客户、资产、问题模板、处理过程。
- 可补充适用场景、处理步骤、注意事项和标签。
- 发布后可在知识库检索。
- 新增运维记录时可引用历史知识。

### US-07 培训负责人记录客户培训和方案

作为培训负责人，我希望记录客户培训计划、培训材料、客户问题和后续动作，这样培训不只是一次会议，而能沉淀为客户经营和方案复用资产。

验收标准：

- 可为客户或项目创建培训计划。
- 可上传培训材料或选择方案库模板。
- 可记录参训人员、客户问题和培训反馈。
- 客户问题可生成 FAQ 知识条目。
- 如果培训反馈出现新需求，可生成商机信号或人工补录商机。

### US-08 区域负责人查看客户作战地图

作为区域负责人，我希望进入系统后能按地市、行业、客户等级、负责人和健康状态查看全区客户，这样我能判断哪些客户需要投入资源、哪些区域存在覆盖空白。

验收标准：

- 首页或区域作战地图可按地市筛选客户。
- 客户点位或列表能显示负责人、客户等级、最近互动、风险标签和有效商机数。
- 可筛选 90 天未互动客户、低数据质量客户、重点客户和预警客户。
- 点击客户可进入客户详情页。
- 可导出区域月度经营复盘数据。

### US-09 客户经理维护权力地图、客户树和协作视图

作为客户经理，我希望记录客户内部关键人、角色、权力等级、亲疏度和立场，并用权力地图、客户树和可视化协作视图呈现客户关系，这样团队交接和客户经营不再依赖个人记忆。

验收标准：

- 客户详情页可新增关键人。
- 关键人可标注角色：决策人、影响人、使用人、信息人、反对者。
- 可记录职务、所属部门、联系方式、亲疏度、立场、影响力、决策角色、关键诉求和备注。
- 可在权力地图中查看和维护关键人节点、上下级关系、影响路径、支持/反对立场和决策链路。
- 可在客户树中维护集团、上下级单位、关联单位、供应链、竞争关系和人事流动。
- 可在可视化协作视图中对客户、关键人、关系线、客户树节点、拜访计划或商机添加评论、标注、协作任务和 @ 成员。
- 重点客户缺少决策人或影响人时，系统提示关系链不完整。
- 关键人信息变更后写入操作日志。

### US-10 客户经理制定拜访计划并回填互动

作为客户经理，我希望为重点客户制定拜访计划，并在拜访、电话、微信、会议或培训后回填互动记录，这样客户关系维护动作可复盘、可提醒、可交接。

验收标准：

- 可创建拜访计划，填写客户、对象、目的、计划时间、准备材料和预期结果。
- 拜访计划可关联商机、项目、培训计划或客户风险。
- 互动完成后可回填互动类型、对象、关键内容、客户反馈、下一步动作和下次跟进时间。
- 拜访计划逾期未完成时提醒负责人。
- 互动记录保存后更新客户最近互动时间和关系亲疏度。

## 10. 规则体系

### 10.1 字段拆分

必须拆分两个字段：

- `signal_type`：商机/风险类型。
- `priority`：处理优先级。

不要把“类型”和“优先级”绑定。安全风险、新需求、竞品风险都可能因为客户等级、资产重要性、到期时间和重复发生情况而变成不同优先级。

### 10.2 MVP 规则表

| 规则编号 | 信号类型 | 触发条件 | 默认优先级 | 推荐动作 |
|---|---|---|---|---|
| R001 | 续约 | 维保/授权/证书到期 <= 30 天，可信度为 contract/confirmed | P0 | 联系客户确认续约或替代方案 |
| R002 | 续约 | 维保/授权/证书到期 <= 60 天，可信度为 imported/manual | P1 | 先核实到期日，再安排沟通 |
| R003 | 扩容 | 文本包含 CPU 高、内存不足、磁盘满、并发高、响应慢 | P1 | 安排技术诊断，判断扩容/优化方案 |
| R004 | 性能优化 | 同一客户同一资产 30 天内出现 3 次性能问题 | P1 | 输出问题分析，建议拜访 |
| R005 | 安全加固 | 文本包含漏洞、弱口令、被攻击、病毒、备份失败、日志审计 | P1 | 安排安全检查或加固方案 |
| R006 | 新需求 | 文本包含能不能、希望、想要、有没有、是否支持、领导要求 | P2 | 客户经理确认场景与预算 |
| R007 | 服务升级 | 文本包含响应慢、服务不及时、驻场、巡检频率 | P2 | 判断服务包升级可能 |
| R008 | 竞品/流失 | 文本包含竞品、替换、别家、重新招标 | P2/P3 | 主管或客户经理确认真实意图 |

### 10.3 优先级计算

优先级由以下因素共同决定：

- 到期时间。
- 客户等级。
- 资产重要性。
- 是否重复发生。
- 是否客户主动表达需求。
- 是否存在明确预算或采购计划。
- 数据可信度。

建议公式：

```text
priority_score =
  到期紧迫度 * 30
+ 客户等级 * 20
+ 资产重要性 * 20
+ 重复发生 * 15
+ 主动需求 * 10
+ 数据可信度 * 5
```

映射：

| 分数 | 优先级 |
|---:|---|
| 80-100 | P0 |
| 60-79 | P1 |
| 40-59 | P2 |
| 0-39 | P3 |

## 11. 商机状态机与 SLA

### 11.1 标准状态

| 状态 | 说明 | 可执行动作 | SLA |
|---|---|---|---|
| 待分配 | 信号已生成但没有负责人 | 分配负责人、关闭、误报 | P0/P1 24h |
| 待跟进 | 已有负责人，尚未开始处理 | 开始跟进、转派、误报、暂缓 | P0/P1 48h |
| 跟进中 | 负责人已开始联系客户 | 新增跟进、确认商机、放弃、暂缓 | 7 天 |
| 已确认商机 | 客户有明确意向 | 创建变更/项目、继续跟进、关闭 | 14 天 |
| 已转化 | 已进入变更/立项/报价流程 | 查看关联对象 | 无 |
| 已放弃 | 客户无意向或暂不推进 | 归档、恢复 | 无 |
| 误报 | 规则识别错误 | 进入误报反馈 | 30 天观察 |
| 已失效 | 长期未处理或未执行 | 恢复、归档 | 无 |

### 11.2 必填字段

| 动作 | 必填字段 |
|---|---|
| 开始跟进 | 下次跟进时间 |
| 确认商机 | 意向强度、预计金额区间、预计成交时间、下一步动作 |
| 放弃 | 放弃原因 |
| 暂缓 | 暂缓原因、重新提醒时间 |
| 误报 | 误报类型、说明 |
| 转派 | 新负责人、转派原因 |

## 13. 客户风险标签

MVP 不做复杂健康度分数，先做轻量风险标签。

| 标签 | 触发规则 |
|---|---|
| 健康 | 近 30 天无高优先级信号，无到期风险 |
| 关注 | 有 P2 信号，或 60 天内有到期资产 |
| 预警 | 有 P0/P1 信号，或 30 天内到期，或重复故障 |
| 商机 | 已确认商机或存在明确客户需求 |

客户详情页展示：

- 当前标签。
- 触发原因。
- 最近 3 条相关记录。
- 推荐动作。

## 14. 通知策略

### 14.1 通知渠道

MVP 优先级：

1. 系统内通知。
2. 企业微信机器人。
3. 邮件。
4. 短信暂缓，仅用于真正紧急的 P0。

### 14.2 通知规则

| 事件 | 通知对象 | 渠道 | 频率 |
|---|---|---|---|
| P0 信号生成 | 负责人、主管 | 系统内 + 企业微信 | 立即 |
| P1 信号生成 | 负责人 | 系统内 + 企业微信 | 立即 |
| P2 信号生成 | 负责人 | 系统内 | 每日汇总 |
| 无负责人 P0/P1 | 主管 | 系统内 + 企业微信 | 立即 |
| 48h 未处理 | 负责人、主管 | 系统内 + 企业微信 | 一次 |
| 跟进到期 | 负责人 | 系统内 | 每日 |
| 到期资产 30 天 | 负责人 | 系统内 + 企业微信 | 每周汇总 |

### 14.3 防打扰原则

- 同一客户同一类型信号 7 天内合并提醒。
- P2/P3 不实时打扰，只做汇总。
- 已确认跟进的信号不重复推送，只按下次跟进时间提醒。

## 15. 权限与数据安全

### 15.1 数据范围

| 角色 | 客户范围 | 商机范围 |
|---|---|---|
| 运维工程师 | 自己负责或参与服务的客户 | 只看自己提交记录生成的信号摘要 |
| 项目经理 | 自己负责项目关联客户 | 自己负责客户的信号 |
| 客户经理/CSM | 分配给自己的客户 | 自己负责客户的信号 |
| 主管 | 本部门客户 | 本部门全部信号 |
| 管理员 | 全部 | 全部 |

### 15.2 敏感字段

默认脱敏：

- 合同金额。
- 联系人手机号。
- 客户原话中的账号、密码、IP、密钥。
- 日志、截图附件中的敏感信息。

原文查看权限：

- 普通负责人：查看脱敏文本。
- 主管：可查看原文。
- 管理员：可查看原文与审计日志。

MVP 阶段不保存原始语音。如后续必须保存，默认 30 天后自动删除。

## 17. 指标体系

### 17.1 北极星指标

每月被复用的服务资产数量 + 被持续维护的重点客户关系资产数量。

服务资产包括：有效项目复盘、有效知识条目、有效客户方案、有效培训记录、已确认商机、关键人关系链、有效互动记录和拜访复盘。该指标比单纯“商机数”更适合 v1.7，因为产品目标已经从商机雷达扩展为“项目交付 + 运维知识 + 区域 CRM + 客户经营 + 培训方案”的服务经营平台。

## 19. 风险与依赖

| 风险 | 描述 | 缓解措施 |
|---|---|---|
| 运维记录录入摩擦高 | 运维人员可能不愿额外填写信息 | 模板点选、20 秒完成、不做考核、每周复盘选项 |
| 负责人不处理信号 | 项目经理/客户经理已有工作压力 | 首页待办、48h 提醒、主管分配池 |
| 规则误报透支信任 | 关键词规则早期可能误报 | P0/P1 宽松但可解释，P2 汇总，保留误报反馈 |
| 资产到期日不准 | 到期数据错误会导致续约信号错报 | 可信度标签、试点前清洗关键资产 |
| 与 M16 依赖不清 | 完整工单系统不在 MVP 范围内 | MVP 内置快速运维记录，不依赖 M16 |
| 客户健康度过重 | 复杂分数会拖慢 MVP | 第一版只做风险标签 |
| CRM 作战地图范围膨胀 | 容易扩展到完整销售自动化、报价、回款、外部数据抓取 | MVP 只做客户分布、关键人、拜访互动、轻量健康和数据质量 |
| 关键人信息敏感 | 联系方式、立场、家庭背景等信息可能引发隐私与合规风险 | 默认脱敏、最小必要字段、权限控制、操作审计，不记录无业务必要的敏感背景 |
| 关系链维护不持续 | 客户经理可能只在上线初期录入，后续不更新 | 低质量提醒、拜访后回填、月度区域复盘，把维护动作嵌入日常流程 |
| 项目管理过重 | 如果直接做甘特图、资源池、成本预算，会拖慢上线 | MVP 只做项目台账、任务、里程碑、风险和交付物 |
| 知识库无人维护 | 知识条目可能变成低质量堆积 | 设置维护人、状态、标签和定期复核 |
| 培训模块偏 LMS 化 | 如果做课程体系、考试、证书，会偏离客户经营目标 | 第一版只记录培训计划、材料、问题和后续动作 |
| 合同/电子签范围膨胀 | 参考上上签容易引入审批、签章、法律存证复杂能力 | MVP 只保留合同到期、归档链接和履约提醒字段 |

## SOP-01 快速运维记录补充流程

| 元数据 | 值 |
|---|---|
| **版本** | 1.1 |
| **负责人** | 运维负责人 / 产品经理 |
| **触发条件** | 每次客户服务、巡检、故障处理或现场沟通后 |
| **预计用时** | 1-3 分钟 |
| **审核周期** | 每月 |

### 目的

确保运维工程师能用低摩擦方式记录客户现场事实，为商机雷达提供可识别的原始信号，同时避免让运维人员承担销售判断职责。

### 范围

- **适用**：运维工程师、项目经理、客户经理/CSM
- **时机**：完成客户巡检、故障处理、咨询支持、变更支持、现场服务后
- **不包含**：完整工单处理、报价、合同、销售推进

### RACI 矩阵

| 步骤 | 执行者（R） | 决策者（A） | 咨询方（C） | 知会方（I） |
|---|---|---|---|---|
| 进入记录入口 | 运维工程师 | 运维工程师 | — | — |
| 选择类型与模板 | 运维工程师 | 运维工程师 | — | — |
| 补充现场事实 | 运维工程师 | 运维工程师 | — | — |
| 保存并触发规则 | 系统 | 系统 | — | 运维工程师 |
| 命中规则后处理 | 客户经理/CSM | 主管 | 运维工程师 | — |

### 流程图

```
[运维完成现场工作]
    ↓
[打开系统 → 运维记录页]
    ↓
[选择客户] → 无客户？→ [提交主管补充]
    ↓
[选择记录类型] 巡检/故障/咨询/变更/现场支持/其他
    ↓
[选择模板] 性能/到期/安全/新需求/投诉/竞品/其他
    ↓
[填写问题描述 + 客户反馈]
    ↓
[保存记录]
    ↓
    ├─→ 命中规则 → [生成商机信号] → SOP-02
    ├─→ 未命中 → [归档为普通记录]
    └─→ 客户/资产缺失 → [标记待补充]
```

### 详细步骤

#### Step 1：进入快速记录入口

- **谁**：运维工程师
- **何时**：完成现场工作后 30 分钟内
- **如何**：
  1. 从首页快捷入口「运维记录」或侧边栏「运维管理 → 运维记录」进入。
  2. 点击「新建工单」（当前界面名称，后续将优化为「快速记录」）。
  3. 选择客户（下拉搜索）。
  4. 如本次涉及具体系统，选择关联资产；不确定则留空。
- **产出**：记录草稿，含客户和资产信息

#### Step 2：选择记录类型与模板

- **谁**：运维工程师
- **何时**：进入记录入口后立即
- **如何**：
  1. 选择记录类型：巡检、故障、咨询、变更、现场支持、其他。
  2. 选择模板（当前版本为类型标签，后续将支持模板点选）：性能问题、磁盘不足、证书到期、客户咨询、安全问题、新需求、投诉、竞品、其他。
  3. 若选「其他」，必须填写不少于 5 个字的说明。
- **产出**：类型和模板标签

#### Step 3：补充现场事实

- **谁**：运维工程师
- **何时**：类型选择后
- **如何**：
  1. 填写问题描述，记录系统现象、客户反馈或现场观察。
  2. 如有客户原话，填写在描述字段（当前版本无独立「客户反馈」字段，后续将拆分）。
  3. **不判断**是否为商机，不填写销售结论。
- **产出**：含事实描述的运维记录

#### Step 4：保存并触发规则

- **谁**：系统（自动）+ 运维工程师（确认保存）
- **何时**：点击保存后
- **如何**：
  1. 运维工程师点击「保存」或「确定」。
  2. 系统生成运维记录（当前为工单编号）。
  3. 系统自动进入规则识别流程（当前为手动录入阶段，后续接入规则引擎）。
- **产出**：已保存的运维记录，可能触发商机信号

### 检查清单

- [ ] 已选择客户
- [ ] 已选择记录类型
- [ ] 已填写问题描述（5 字以上）
- [ ] 已保存记录
- [ ] 如涉及资产，已关联或标记待补充
- [ ] 未在描述中填写销售判断

### 异常处理

| 异常场景 | 处理方式 | 升级路径 |
|---|---|---|
| 找不到客户 | 客户档案未创建或无权限 | 联系主管或管理员补齐客户档案 |
| 找不到资产 | 资产台账未维护 | 先保存记录，后续在资产台账补充 |
| 不知道选哪个类型 | 场景不明确 | 选择「其他」，填写简短说明 |
| 记录包含敏感信息（账号/密码/IP） | 涉及隐私/安全 | 系统标记 sensitive_flag，按脱敏规则处理 |
| 记录内容被判定为销售结论 | 违反记录原则 | 主管提醒，重新记录为事实描述 |

## SOP-02 商机信号识别与生成流程

| 元数据 | 值 |
|---|---|
| **版本** | 1.1 |
| **负责人** | 产品/运营管理员 |
| **触发条件** | 每次新增运维记录、资产到期扫描、规则配置变更后 |
| **预计用时** | 系统自动；人工复核 1-5 分钟 |
| **审核周期** | 每周 |

### 目的

将运维记录、资产到期和客户反馈转化为可解释的商机信号，确保每条信号都能说明触发原因、优先级和推荐动作。

### 范围

- **适用**：规则引擎、产品/运营管理员、客户经理/CSM、主管
- **时机**：新增运维记录、资产到期扫描、规则配置变更后
- **不包含**：AI 自动判断、复杂预测模型、CRM 销售阶段管理

### RACI 矩阵

| 步骤 | 执行者（R） | 决策者（A） | 咨询方（C） | 知会方（I） |
|---|---|---|---|---|
| 读取触发源 | 系统 | 系统 | — | 运维工程师 |
| 执行规则匹配 | 系统 | 产品/运营管理员 | — | — |
| 计算优先级 | 系统 | 系统 | — | — |
| 生成商机信号 | 系统 | 系统 | — | 负责人、主管 |
| 复核误报 | 负责人 | 主管 | 产品/运营管理员 | — |

### 流程图

```
[运维记录保存] / [资产到期扫描]
    ↓
[系统读取触发源]
    ↓
[提取字段：客户/资产/类型/描述/到期日/可信度]
    ↓
[规则匹配]
    ├─→ 到期类规则（R001/R002）
    ├─→ 性能/扩容关键词（R003/R004）
    ├─→ 安全关键词（R005）
    ├─→ 新需求关键词（R006）
    ├─→ 服务升级关键词（R007）
    └─→ 竞品关键词（R008）
    ↓
[计算优先级分数]
    到期紧迫度*30 + 客户等级*20 + 资产重要性*20 + 重复发生*15 + 主动需求*10 + 可信度*5
    ↓
[映射 P0/P1/P2/P3]
    ↓
[生成商机信号]
    ├─→ 同客户+同资产+同类型未关闭 → [合并提醒]
    ├─→ 无负责人 → [待分配状态]
    └─→ 有负责人 → [待跟进状态]
    ↓
[通知推送]
    ├─→ P0/P1：立即通知负责人+主管（系统内+企业微信）
    ├─→ P2：系统内通知或每日汇总
    └─→ P3：不主动推送，进入低优先级列表
```

### 详细步骤

#### Step 1：读取触发源

- **谁**：系统（自动）
- **何时**：运维记录保存或资产到期扫描触发后
- **如何**：
  1. 读取新增运维记录或资产到期数据。
  2. 提取客户、资产、记录类型、模板、描述、客户反馈、到期日、数据可信度。
- **产出**：结构化触发源数据

#### Step 2：执行规则匹配

- **谁**：系统（自动）
- **何时**：读取触发源后
- **如何**：
  1. 匹配到期类规则（<=30天 P0 / <=60天 P1）。
  2. 匹配性能/扩容关键词（CPU高、内存不足、磁盘满、并发高、响应慢）。
  3. 匹配安全关键词（漏洞、弱口令、被攻击、病毒、备份失败、日志审计）。
  4. 匹配新需求关键词（能不能、希望、想要、有没有、是否支持、领导要求）。
  5. 匹配服务升级关键词（响应慢、服务不及时、驻场、巡检频率）。
  6. 匹配竞品关键词（竞品、替换、别家、重新招标）。
  7. 判断重复发生：同一客户同一资产 30 天内是否出现多次同类问题。
- **产出**：规则命中列表，含触发规则编号和匹配文本

#### Step 3：计算优先级

- **谁**：系统（自动）
- **何时**：规则匹配后
- **如何**：
  1. 按公式计算 priority_score：到期紧迫度*30 + 客户等级*20 + 资产重要性*20 + 重复发生*15 + 主动需求*10 + 数据可信度*5。
  2. 映射分数：80-100→P0，60-79→P1，40-59→P2，0-39→P3。
  3. 写入 trigger_reason 和 recommended_action。
- **产出**：优先级分数、P0/P1/P2/P3 标签、推荐动作

#### Step 4：生成商机信号

- **谁**：系统（自动）
- **何时**：优先级计算后
- **如何**：
  1. 如命中规则，创建 opportunity_signal。
  2. 如已有同客户、同资产、同类型未关闭信号，合并提醒，不重复生成。
  3. 如无默认负责人，状态设为「待分配」。
  4. 如有默认负责人，状态设为「待跟进」。
- **产出**：商机信号记录，进入商机池

### 检查清单

- [ ] 已读取触发源
- [ ] 已匹配规则
- [ ] 已计算优先级分数
- [ ] 已生成触发原因文本
- [ ] 已生成推荐动作文本
- [ ] 已写入商机池
- [ ] 已按优先级触发通知
- [ ] 重复信号已合并

### 异常处理

| 异常场景 | 处理方式 | 升级路径 |
|---|---|---|
| 应该触发但没有触发 | 关键词缺失或资产数据缺失 | 管理员补充规则或资产字段 → 产品/运营管理员 |
| 重复生成多个相同信号 | 去重规则缺失 | 按客户+资产+类型+未关闭状态合并 |
| 优先级明显过高 | 到期日或客户等级错误 | 修正基础数据并重新计算 |
| 误报过多 | 关键词过宽 | 进入 SOP-05 误报反馈与规则优化流程 |
| 规则引擎离线 | 系统故障 | 转入人工识别模式 → 主管每日巡查 |

## SOP-03 商机分配与跟进流程

| 元数据 | 值 |
|---|---|
| **版本** | 1.1 |
| **负责人** | 客户成功主管 |
| **触发条件** | 商机信号进入「待分配」「待跟进」「跟进中」状态 |
| **预计用时** | 单条 3-10 分钟 |
| **审核周期** | 每周 |

### 目的

确保商机信号生成后有人负责、有人响应、有人记录结果，避免信号停留在列表中无人处理。

### 范围

- **适用**：主管、项目经理、客户经理/CSM
- **时机**：商机信号进入「待分配」「待跟进」「跟进中」状态
- **不包含**：报价、合同签署、正式销售阶段管理

### RACI 矩阵

| 步骤 | 执行者（R） | 决策者（A） | 咨询方（C） | 知会方（I） |
|---|---|---|---|---|
| 处理待分配信号 | 主管 | 主管 | 运维工程师 | 负责人 |
| 开始跟进 | 客户经理/CSM | 客户经理/CSM | — | 主管 |
| 记录跟进结果 | 客户经理/CSM | 客户经理/CSM | — | 主管 |
| 确认商机 | 客户经理/CSM | 主管 | 项目经理 | 区域负责人 |
| 超时升级 | 系统 | 主管 | — | 部门负责人 |

### 流程图

```
[商机信号生成]
    ├─→ 无负责人 → [待分配状态]
    │               ↓
    │           [主管查看待分配列表]
    │               ↓
    │           [按分配规则选择负责人]
    │               ↓
    │           [点击分配]
    │               ↓
    └─→ 有负责人 → [待跟进状态]
                    ↓
            [负责人查看商机池/首页待办]
                    ↓
            [查看触发原因、来源记录、推荐动作]
                    ↓
            [点击「开始跟进」]
                    ↓
            [填写下次跟进时间]
                    ↓
            [状态变为「跟进中」]
                    ↓
            [与客户沟通]
                    ↓
            [新增跟进记录]
                    ↓
            [选择处理结果]
                ├─→ 确认商机 → [填写意向/金额/预计成交时间] → 进入真实业务动作
                ├─→ 放弃 → [填写放弃原因] → 归档
                ├─→ 暂缓 → [填写原因+重新提醒时间] → 状态冻结
                └─→ 误报 → [进入 SOP-05]
```

### 详细步骤

#### Step 1：处理待分配信号

- **谁**：主管
- **何时**：每日上午查看「待分配信号」
- **如何**：
  1. 打开首页「待分配信号」或进入商机池筛选「待分配」。
  2. 查看客户、信号类型、优先级、触发原因。
  3. 按负责人分配规则选择项目经理、客户经理或 CSM。
  4. 点击「分配负责人」。
- **产出**：已分配负责人，负责人收到通知

#### Step 2：负责人开始跟进

- **谁**：项目经理 / 客户经理 / CSM
- **何时**：收到通知后 48 小时内（P0/P1）
- **如何**：
  1. 打开「商机池」或首页待办。
  2. 查看触发原因、来源运维记录和推荐动作。
  3. 点击「开始跟进」。
  4. 填写下次跟进时间（必填）。
- **产出**：状态变为「跟进中」，下次跟进时间已设定

#### Step 3：记录跟进结果

- **谁**：负责人
- **何时**：与客户沟通后
- **如何**：
  1. 新增跟进记录（商机池 → 跟进记录页）。
  2. 填写沟通内容、客户态度、下一步动作。
  3. 根据结果选择：确认商机、放弃、暂缓、误报。
- **产出**：跟进记录，状态变更

#### Step 4：确认商机（若选择确认）

- **谁**：负责人
- **何时**：客户有明确意向后
- **如何**：
  1. 点击「确认商机」。
  2. 填写意向强度、预计金额区间、预计成交时间、下一步动作。
  3. 如预计金额 >= 10 万，同步客户经理/销售负责人。
  4. 后续进入报价、变更、立项或客户拜访（SOP-10）。
- **产出**：已确认商机，进入后续业务动作

### 检查清单

- [ ] 待分配信号已分配负责人
- [ ] 负责人已查看触发原因和来源记录
- [ ] 已点击「开始跟进」
- [ ] 已填写下次跟进时间
- [ ] 已记录跟进结果
- [ ] 已选择最终处理动作（确认/放弃/暂缓/误报）
- [ ] 已确认商机已填写意向和预计金额

### 异常处理

| 异常场景 | 处理方式 | 升级路径 |
|---|---|---|
| 无法判断分配给谁 | 客户归属不清 | 主管按客户阶段临时指定负责人 |
| 负责人长期未处理（>48h） | 工作负载过高或通知未读 | 主管转派或调整优先级 → 部门负责人 |
| 客户无明确意向 | 信号有价值但时机不成熟 | 标记暂缓并设置重新提醒时间 |
| 客户明确无需求 | 商机不成立 | 标记放弃并填写原因 |
| 超过 7 天无进展 | 跟进停滞 | 系统提醒负责人 → 主管介入 |
| 已确认商机 14 天无业务动作 | 商机未推进 | 标记为待复核 → 主管评估 |

## SOP-04 关键资产与到期数据维护流程

| 元数据 | 值 |
|---|---|
| **版本** | 1.1 |
| **负责人** | 运维负责人 / 数据管理员 |
| **触发条件** | 新客户导入、服务合同签署、资产变化、到期日前验证、现场发现数据错误 |
| **预计用时** | 初始导入 30-60 分钟；单条维护 2-5 分钟 |
| **审核周期** | 每月 |

### 目的

保证维保、授权、证书等到期类数据准确，避免续约类 P0/P1 信号因数据错误而误报或漏报。

### 范围

- **适用**：运维工程师、项目经理、管理员
- **时机**：新客户导入、服务合同签署、资产变化、到期日前验证、运维现场发现数据错误
- **不包含**：合同审批、采购流程、资产采购入库

### RACI 矩阵

| 步骤 | 执行者（R） | 决策者（A） | 咨询方（C） | 知会方（I） |
|---|---|---|---|---|
| 初始录入资产 | 运维工程师/管理员 | 运维负责人 | 客户经理 | 系统 |
| 每月验证到期数据 | 运维工程师 | 运维负责人 | — | 主管 |
| 现场修正数据 | 运维工程师/项目经理 | 项目经理/主管 | — | 系统 |
| 信号触发复核 | 系统 | 系统 | 运维工程师 | 负责人 |

### 流程图

```
[新客户导入/合同签署/资产变化]
    ↓
[进入资产台账页]
    ↓
[新增/编辑资产]
    ├─→ 填写：资产编号、名称、分类、型号、厂商、位置、状态
    ├─→ 填写：采购日期、维保到期日、原值
    └─→ 设置数据可信度：imported / manual / contract / confirmed
    ↓
[保存资产]
    ↓
[系统每月生成 90 天到期清单]
    ↓
[运维工程师逐条确认/修改]
    ↓
[已确认数据标记为 confirmed]
    ↓
    ├─→ confirmed + 30 天内到期 → [P0 信号]
    ├─→ imported/manual + 60 天内到期 → [P1 信号，推荐核实]
    └─→ 无资产数据 → [「资产待补充」提醒，不生成 P0]
```

### 详细步骤

#### Step 1：初始录入关键资产

- **谁**：运维工程师 / 管理员
- **何时**：新客户导入、合同签署后
- **如何**：
  1. 进入「运维管理 → 资产台账」页。
  2. 点击「新增资产」。
  3. 填写资产编号（必填）、资产名称（必填）、分类（服务器/网络/存储/安全/办公）、型号、厂商、位置、状态（正常/维修中/闲置/报废）。
  4. 填写采购日期、维保到期日（关键）、原值。
  5. 设置数据可信度：imported（导入）、manual（手动）、contract（合同）、confirmed（已确认）。
  6. 保存。
- **产出**：已录入资产，进入资产台账

#### Step 2：每月验证到期数据

- **谁**：运维工程师
- **何时**：每月第一周
- **如何**：
  1. 系统生成未来 90 天到期资产清单（当前为手动筛选，后续将自动化）。
  2. 运维工程师逐条确认或修改到期日。
  3. 已确认数据标记为 confirmed。
- **产出**：已验证的到期数据清单

#### Step 3：现场发现数据错误时修正

- **谁**：运维工程师 / 项目经理
- **何时**：现场发现资产信息与系统不符
- **如何**：
  1. 在「客户详情」或「资产台账」中找到该资产。
  2. 点击「编辑」。
  3. 修改字段并填写修正原因。
  4. 项目经理或主管确认后生效。
- **产出**：已修正的资产数据，操作日志记录

### 检查清单

- [ ] 客户关键资产已录入
- [ ] 到期日已填写（维保、授权、证书）
- [ ] 数据可信度已标记
- [ ] 每月验证清单已处理
- [ ] 修正记录有原因和确认人
- [ ] 无资产客户已标记「资产待补充」

### 异常处理

| 异常场景 | 处理方式 | 升级路径 |
|---|---|---|
| 客户无资产数据 | 历史数据未整理 | 标记资产待补充，先不生成 P0 → 运维负责人安排录入 |
| 到期日冲突（合同/证书/现场不一致） | 信息来源不一致 | 以合同或主管确认为准 → 数据管理员仲裁 |
| 已续约仍提醒 | 系统未更新到期日 | 提交修正并关闭旧信号 |
| 同一系统多个到期日 | 维保、授权、证书并存 | 分别记录，取最近日期触发提醒 |
| 资产状态与实际情况不符 | 系统未更新 | 现场修正 → 项目经理确认 |

## SOP-05 误报反馈与规则优化流程

| 元数据 | 值 |
|---|---|
| **版本** | 1.1 |
| **负责人** | 产品/运营管理员 |
| **触发条件** | 负责人判断信号不应出现，或触发类型/优先级明显错误 |
| **预计用时** | 单条 1-3 分钟；每周复盘 30 分钟 |
| **审核周期** | 每周 |

### 目的

让业务人员能快速反馈错误信号，让产品/运营管理员持续优化规则，避免误报透支用户信任。

### 范围

- **适用**：项目经理、客户经理/CSM、主管、产品/运营管理员
- **时机**：负责人判断信号不应出现，或触发类型/优先级明显错误
- **不包含**：客户暂时无需求但规则识别合理的「已放弃」场景

### RACI 矩阵

| 步骤 | 执行者（R） | 决策者（A） | 咨询方（C） | 知会方（I） |
|---|---|---|---|---|
| 标记误报 | 负责人 | 负责人 | — | 系统 |
| 系统记录证据 | 系统 | 系统 | — | 产品/运营管理员 |
| 每周复盘误报 | 产品/运营管理员 | 产品/运营管理员 | 主管 | 产品经理 |
| 更新规则配置 | 产品/运营管理员 | 产品经理 | — | 全员 |

### 流程图

```
[负责人发现信号错误]
    ↓
[进入商机详情页]
    ↓
[点击「误报」]
    ↓
[选择误报类型]
    ├─→ 规则错误
    ├─→ 时机不对
    ├─→ 不相关
    └─→ 基础数据错误
    ↓
[填写简短说明]
    ↓
[系统保存证据]
    ├─→ trigger_rule
    ├─→ source_text

    ├─→ marked_by

    ↓
[信号状态变更为「误报」]
    ↓

    ↓

    ↓

    ↓

```

### 详细步骤

#### Step 1：标记误报

- **谁**：负责人
- **何时**：发现信号不应出现或明显错误时
- **如何**：
  1. 在商机详情中点击「误报」。
  2. 选择误报类型：规则错误、时机不对、不相关、基础数据错误。
  3. 填写简短说明（如：「客户刚续约，到期日未更新」）。
- **产出**：误报标记，含类型和说明

#### Step 2：系统记录误报证据

- **谁**：系统（自动）
- **何时**：误报标记后
- **如何**：
  1. 保存 trigger_rule、source_text、source_record_id、marked_by、marked_at。
  2. 信号状态变更为「误报」。
  3. 写入 signal_event_log。
- **产出**：完整误报证据链

#### Step 3：每周复盘误报

- **谁**：产品/运营管理员
- **何时**：每周固定时间（建议周五下午）
- **如何**：
  1. 导出本周误报列表（当前为手动统计，后续将支持报表导出）。
  2. 按规则编号和误报类型聚合。
  3. 判断是否需要调整关键词、阈值、优先级或数据校验。
- **产出**：误报分析摘要

#### Step 4：更新规则配置

- **谁**：产品/运营管理员
- **何时**：复盘后，需要调整时
- **如何**：
  1. 修改 rule_config（当前为后端配置，后续将支持前端配置页）。
  2. 记录修改原因。
  3. 对最近 30 天数据做简单回看，确认不会明显增加漏检。
- **产出**：已更新的规则配置，修改记录

### 检查清单

- [ ] 已选择误报类型
- [ ] 已填写说明
- [ ] 已记录触发规则和来源文本
- [ ] 已进入每周复盘
- [ ] 规则调整已记录原因
- [ ] 已验证不增加漏检

### 异常处理

| 异常场景 | 处理方式 | 升级路径 |
|---|---|---|
| 用户把无需求标成误报 | 不理解误报定义 | 在界面提示「无需求请选择放弃/暂缓」→ 培训 |
| 误报集中在某类关键词 | 关键词过宽 | 收紧关键词或增加排除词 |
| 调整规则后漏检增加 | 收紧过度 | 回滚规则并补充分层优先级 → 产品经理 |
| 无法追溯来源文本 | 日志缺失 | 补齐 source_record_id 和 trigger_rule → 研发 |

## SOP-06 项目创建与流程节点维护流程

| 元数据 | 值 |
|---|---|
| **版本** | 1.1 |
| **负责人** | 项目经理 |
| **触发条件** | 新项目立项、项目分类变化、流程节点调整、任务拆分、客户交流汇报、项目会议或项目风险出现 |
| **预计用时** | 项目创建 10-15 分钟；节点维护 5-10 分钟/节点 |
| **审核周期** | 每周（项目周会） |

### 目的

确保项目从前期调研到运维移交的全流程都有统一记录，并能按项目金额、类型和交付流程进行管理。每次客户交流、项目例会和阶段汇报都应沉淀会议纪要，避免项目过程只存在于群消息和个人表格中。

### 范围

- **适用**：项目经理、项目成员、实施、运维、客户经理、客户侧干系人
- **时机**：项目启动、客户交流/会议后、每周复盘
- **不包含**：合同审批、采购付款、正式验收法律流程

### RACI 矩阵

| 步骤 | 执行者（R） | 决策者（A） | 咨询方（C） | 知会方（I） |
|---|---|---|---|---|
| 创建项目 | 项目经理 | 主管 | 客户经理 | 项目成员 |
| 配置流程节点 | 项目经理 | 项目经理 | 实施/运维 | 客户侧干系人 |
| 拆分任务 | 项目经理/节点负责人 | 项目经理 | — | 成员 |
| 更新进度/上传交付物 | 项目成员 | 项目经理 | — | — |
| 设备到货确认 | 采购/实施 | 项目经理 | 运维 | 客户 |
| 会议纪要生成 | 会议负责人 | 项目经理 | — | 参会人 |
| 风险登记 | 项目经理 | 主管 | — | 成员 |
| 项目复盘 | 项目经理 | 主管 | 团队 | 客户 |

### 流程图

```
[项目立项]
    ↓
[创建项目]
    ├─→ 客户、金额、金额分层、类型、阶段、计划周期、成员
    ↓
[选择/配置流程节点]
    ├─→ 默认模板：前期调研 → 方案沟通 → 合同评审 → 售前交接 → 采购 → 需求 → 设备到货 → 方案功能确认 → 安装调试 → 沟通交付 → 变更 → 测试 → 培训 → 初验 → 终验 → 运维移交
    ↓
[为每个节点设置负责人/计划时间/状态/确认要求/输出产物]
    ↓
[拆分任务 → 明确负责人/截止时间/交付物]
    ↓
[项目成员按节点更新进度/上传交付物]
    ↓
    ├─→ 设备到货 → 登记型号/数量/序列号/授权/签收/照片
    ├─→ 方案功能确认 → 记录功能范围/边界/验收口径/客户意见
    ├─→ 安装调试 → 记录位置/连通/配置/联调/整改
    ├─→ 客户培训 → 记录对象/内容/材料/签到/问题/反馈
    ├─→ 客户交流/会议 → 上传录音/录入内容 → 生成纪要 → 提取待办/风险/反馈 → 同步干系人
    ├─→ 风险出现 → 登记风险/影响/处理人/截止时间
    ↓
[节点确认]
    ├─→ 需要确认的节点 → 确认人填写意见 → 进入「已确认」
    └─→ 无需确认 → 直接进入下一节点
    ↓
[项目验收/关闭]
    ├─→ 检查任务/节点/纪要/交付物完整性
    └─→ 发起复盘 → 沉淀问题/经验/方案/知识
```

### 详细步骤

#### Step 1：创建项目

- **谁**：项目经理
- **何时**：项目立项后
- **如何**：
  1. 进入「项目交付 → 项目概览」，点击「新增项目」。
  2. 填写项目编号（必填）、项目名称（必填）、客户（必填）、阶段（需求分析/方案设计/开发实施/测试验收/交付运维）、状态（正常/进行中/关注/风险/延期/已完成/待验收）。
  3. 填写负责人（必填）、计划时间（RangePicker）、进度（0-100）。
  4. 保存。
- **产出**：已创建项目，进入项目列表

#### Step 2：配置项目节点（时间线）

- **谁**：项目经理
- **何时**：项目创建后
- **如何**：
  1. 进入「项目交付 → 项目节点」。
  2. 选择项目。
  3. 点击「添加节点」，填写节点标题、描述、负责人、计划日期。
  4. 节点状态：未开始（灰）/进行中（蓝）/已完成（绿）/已延期（红）。
  5. 按实际项目进展更新节点状态。
- **产出**：项目时间线，可视化进度

#### Step 3：维护会议记录

- **谁**：会议负责人
- **何时**：每次客户交流、项目例会、阶段汇报后
- **如何**：
  1. 进入「项目交付 → 会议管理」，点击「新增会议」。
  2. 填写会议主题（必填）、所属项目（Select）、会议类型（项目启动/方案评审/周例会/验收会议）、时间范围、地点、参会人。
  3. 保存。
  4. 点击「查看摘要」查看会议纪要（当前为手动录入文本，后续将支持录音转写）。
- **产出**：会议记录，可查看摘要

#### Step 4：项目风险处理

- **谁**：项目经理
- **何时**：风险出现时
- **如何**：
  1. 登记风险描述、影响范围、处理人、状态。
  2. 设置截止时间。
  3. 跟踪至关闭。
- **产出**：风险记录，已处理/关闭

#### Step 5：项目复盘

- **谁**：项目经理
- **何时**：项目结束后
- **如何**：
  1. 检查任务、节点、纪要、交付物是否完整。
  2. 发起复盘，沉淀问题、经验、方案模板和可复用知识。
  3. 复盘内容进入知识库。
- **产出**：项目复盘文档，知识条目

### 检查清单

- [ ] 项目已关联客户
- [ ] 项目已填写金额、阶段、负责人
- [ ] 已配置项目流程节点
- [ ] 每个节点已设置负责人、状态和计划时间
- [ ] 关键交付物已上传或关联到节点
- [ ] 每次关键会议已生成并确认纪要
- [ ] 风险问题已登记处理人和截止时间
- [ ] 项目关闭前已完成复盘

### 异常处理

| 异常场景 | 处理方式 | 升级路径 |
|---|---|---|
| 节点负责人离职/变更 | 重新分配负责人 | 项目经理调整 → 主管备案 |
| 节点延期超过计划 30% | 风险升级 | 登记风险 → 主管介入 |
| 客户拒绝确认节点 | 沟通记录 | 会议纪要备注 → 主管协调 |
| 交付物缺失 | 补齐或说明跳过原因 | 项目经理负责 → 验收前检查 |
| 会议纪要待办未执行 | 超时提醒 | 系统提醒 → 项目经理跟进 |

## SOP-07 运维知识沉淀流程

| 元数据 | 值 |
|---|---|
| **版本** | 1.1 |
| **负责人** | 运维负责人 |
| **触发条件** | 运维记录关闭、巡检发现典型问题、项目复盘完成 |
| **预计用时** | 单条知识 10-15 分钟；每周复盘 30 分钟 |
| **审核周期** | 每周 |

### 目的

把运维记录、巡检异常和项目复盘中的可复用经验沉淀为知识库，形成公司级服务资产。

### 范围

- **适用**：运维工程师、项目经理、产品/运营管理员
- **时机**：运维记录关闭、巡检发现典型问题、项目复盘完成
- **不包含**：非技术类行政流程、个人笔记

### RACI 矩阵

| 步骤 | 执行者（R） | 决策者（A） | 咨询方（C） | 知会方（I） |
|---|---|---|---|---|
| 标记知识候选 | 系统 | 系统 | — | 运维工程师 |
| 沉淀知识 | 运维工程师 | 运维负责人 | — | 团队 |
| 审核发布 | 运维负责人 | 运维负责人 | 产品/运营管理员 | 全员 |
| 知识引用 | 运维工程师/项目经理 | — | 知识维护人 | — |
| 知识维护 | 知识维护人 | 运维负责人 | — | — |

### 流程图

```
[运维记录关闭/巡检完成/项目复盘]
    ↓
[系统标记「知识候选」]
    ├─→ 记录类型为故障/典型问题/复盘
    ├─→ 处理结果明确可复用
    └─→ 复用价值高
    ↓
[运维工程师点击「沉淀知识」]
    ↓
[系统自动带入]
    ├─→ 客户、资产、问题模板、处理过程、结果
    ↓
[维护人补充]
    ├─→ 适用场景
    ├─→ 处理步骤（详细）
    ├─→ 注意事项
    ├─→ 标签（客户/行业/资产/问题模板）
    └─→ 附件（配置截图、日志、脚本）
    ↓
[运维负责人审核]
    ├─→ 通过 → 发布为「已发布」
    └─→ 不通过 → 退回修改/标记「草稿」
    ↓
[知识进入知识库]
    ↓
[后续记录/项目/培训可引用]
    ↓
[知识多次被反馈无效] → [状态变更为「需更新」] → [维护人修订]
```

### 详细步骤

#### Step 1：标记知识候选

- **谁**：系统（自动，后续实现）/ 运维工程师（手动标记）
- **何时**：运维记录关闭、巡检完成、项目复盘后
- **如何**：
  1. 系统根据记录类型、处理结果和复用价值标记「知识候选」（当前为手动判断）。
  2. 运维工程师在记录详情页判断是否为可复用经验。
- **产出**：知识候选标记

#### Step 2：沉淀知识

- **谁**：运维工程师
- **何时**：标记知识候选后
- **如何**：
  1. 从记录中点击「沉淀知识」（当前为进入知识库手动创建）。
  2. 系统自动带入客户、资产、问题模板、处理过程和结果。
  3. 维护人补充适用场景、处理步骤、注意事项、标签和附件。
- **产出**：知识条目草稿

#### Step 3：审核发布

- **谁**：运维负责人
- **何时**：知识条目提交后
- **如何**：
  1. 审核内容完整性、可复用性、准确性。
  2. 通过则发布为「已发布」。
  3. 不通过则退回修改或标记「草稿」。
- **产出**：已发布知识条目

#### Step 4：知识引用与维护

- **谁**：运维工程师/项目经理（引用），知识维护人（维护）
- **何时**：遇到类似问题时引用；知识过期时维护
- **如何**：
  1. 新增运维记录或培训问题时，检索知识库并引用。
  2. 如知识多次被反馈无效，状态变更为「需更新」，维护人修订。
- **产出**：知识引用记录，更新后的知识条目

### 检查清单

- [ ] 来源记录完整
- [ ] 处理步骤可复用
- [ ] 已填写适用场景和标签
- [ ] 已指定维护人
- [ ] 已通过审核发布
- [ ] 知识被后续引用或反馈

### 异常处理

| 异常场景 | 处理方式 | 升级路径 |
|---|---|---|
| 运维工程师不愿沉淀知识 | 工作负载或习惯问题 | 低摩擦入口（一键生成草稿）→ 主管鼓励 |
| 知识质量低 | 处理步骤不清晰 | 退回修改 → 运维负责人指导 |
| 知识过时 | 技术/环境变化 | 标记「需更新」→ 维护人修订 |
| 知识无人引用 | 标签不准确或场景太窄 | 优化标签和场景描述 |
| 敏感信息泄露 | 知识含密码/IP/账号 | 敏感审核 → 脱敏处理 → 管理员 |

## SOP-08 客户培训与方案沉淀流程

| 记录培训结果 | 培训负责人 | 培训负责人 | — | 团队 |
| 沉淀 FAQ/商机 | 培训负责人 | 客户经理 | 运维工程师 | 主管 |

### 流程图

```
    ↓
[创建培训计划]
    ├─→ 关联客户、项目
    ├─→ 培训主题、类型（技术培训/产品培训/销售培训/管理培训）

    ├─→ 参训人员
    ├─→ 客户问题
    ├─→ 培训反馈
    ├─→ 后续动作
    └─→ 状态：已完成
    ↓
[客户问题整理为 FAQ]
    ├─→ 进入知识库
    ↓
[培训中出现新需求/扩容/安全/服务升级]
    ├─→ 生成商机信号 或 人工补录
    ↓
[培训记录沉淀到客户详情页]
    ↓
[进入客户简报素材]
```

### 详细步骤

#### Step 1：创建培训计划

- **谁**：培训负责人
- **何时**：确定需要培训后
- **如何**：
  1. 进入「知识分享 → 培训计划」，点击「新建计划」。
  2. 填写培训标题（必填）、类型（技术培训/产品培训/销售培训/管理培训）、培训对象（必填）、开始日期、结束日期、讲师（必填）。
  3. 状态默认「计划中」。
  4. 保存。
- **产出**：培训计划，状态「计划中」

#### Step 2：准备培训材料

- **谁**：培训负责人
- **何时**：培训执行前
- **如何**：
  1. 选择或上传培训材料。
  2. 可关联方案库中的方案模板。
  3. 准备客户简报（如需）。
- **产出**：培训材料就绪

#### Step 3：执行培训并更新状态

- **谁**：培训负责人
- **何时**：培训当天
- **如何**：
  1. 培训开始前，点击「开始」，状态变为「进行中」。
  2. 执行培训。
  3. 培训结束后，点击「完成」，状态变为「已完成」。
- **产出**：培训已完成，状态更新

#### Step 4：记录培训结果

- **谁**：培训负责人
- **何时**：培训完成后 48 小时内
- **如何**：
  1. 记录参训人员、客户问题、培训反馈、后续动作。
  2. 客户问题可整理为 FAQ 知识条目。
  3. 如培训中出现新需求、扩容、安全加固、服务升级等信号，生成商机信号或人工补录。
- **产出**：培训记录，FAQ 条目，可能的商机信号

### 检查清单

- [ ] 培训计划已关联客户和项目
- [ ] 培训材料已上传或关联方案库
- [ ] 已记录参训人员
- [ ] 已记录客户问题和后续动作
- [ ] 可复用问题已沉淀为 FAQ
- [ ] 可能商机已进入商机池或人工补录
- [ ] 培训记录已沉淀到客户详情页

### 异常处理

| 异常场景 | 处理方式 | 升级路径 |
|---|---|---|
| 客户未参加培训 | 记录原因 | 客户经理跟进 → 重新安排 |
| 培训中出现客户投诉 | 现场安抚 | 记录投诉 → 项目经理介入 → 主管 |
| 培训材料不符合客户场景 | 客户反馈 | 调整材料 → 重新培训 |
| 客户问题超出培训范围 | 记录问题 | 转入运维记录或商机信号 |
| 培训后客户仍不会操作 | 培训效果差 | 补充培训 → 录制视频教程 |

## SOP-09 区域客户作战地图维护流程

| 元数据 | 值 |
|---|---|
| **版本** | 1.1 |
| **负责人** | 区域负责人 |
| **触发条件** | 新客户导入、客户归属调整、月度区域复盘、重点客户状态变化 |
| **预计用时** | 每周检查 30 分钟；每月正式复盘 2-4 小时 |
| **审核周期** | 每月 |

### 目的

保证区域作战地图中的地市、行业、客户等级、负责人、风险状态和数据质量持续准确，让客户分布和经营状态可视化、可复盘。

### 范围

- **适用**：区域负责人、客户经理/CSM、主管、产品/运营管理员
- **时机**：新客户导入、客户归属调整、月度区域复盘、重点客户状态变化
- **不包含**：客户详细经营方案、报价策略

### RACI 矩阵

| 步骤 | 执行者（R） | 决策者（A） | 咨询方（C） | 知会方（I） |
|---|---|---|---|---|
| 导入客户基础数据 | 管理员/客户经理 | 区域负责人 | — | 主管 |
| 生成客户地图 | 系统 | 系统 | — | 区域负责人 |
| 处理待补充客户 | 客户经理 | 区域负责人 | — | 主管 |
| 月度区域复盘 | 区域负责人 | 主管 | 客户经理 | 部门负责人 |
| 导出复盘数据 | 系统 | 区域负责人 | — | 团队 |

### 流程图

```
[新客户导入/客户归属调整/状态变化]
    ↓
[导入客户基础数据]
    ├─→ 客户名称、地市、行业、等级、负责人、服务状态
    ↓
[系统生成客户地图和列表]
    ├─→ 按地市、行业、等级、负责人生成
    ↓
[系统识别待补充客户]
    ├─→ 缺失地市、负责人、客户等级、关键人、最近互动
    ↓
[客户经理每周处理待补充客户]
    ├─→ 补齐字段 或 提交无法补齐原因
    ↓
[区域负责人每月复盘]
    ├─→ 地市覆盖
    ├─→ 重点客户
    ├─→ 预警客户
    ├─→ 有效商机
    ├─→ 拜访达成
    ├─→ 数据质量
    ↓
[对问题指定处理人和完成时间]
    ↓
[导出月度复盘数据]
    ↓
[作为下月客户经营计划输入]
```

### 详细步骤

#### Step 1：导入客户基础数据

- **谁**：管理员或客户经理
- **何时**：新客户导入、归属调整时
- **如何**：
  1. 进入「客户资产 → 客户档案」，点击「新增客户」。
  2. 填写客户名称（必填）、类型（政府机关/医疗机构/金融机构/教育机构/企业）、城市（必填）、行业（必填）。
  3. 设置客户分级（A战略/B重要/C普通/D潜在）、健康度（健康/正常/关注/风险）、负责人（必填）。
  4. 保存。
- **产出**：客户档案，进入客户地图

#### Step 2：查看区域地图

- **谁**：区域负责人、客户经理
- **何时**：日常查看或月度复盘
- **如何**：
  1. 进入「客户资产 → 区域地图」。
  2. 左侧显示网格地图（宁夏 5 地市：银川、石嘴山、吴忠、中卫、固原）。
  3. 点击城市气泡，右侧显示该城市客户列表。
  4. 可查看区域分布统计卡片。
- **产出**：区域客户分布可视化

#### Step 3：处理待补充客户

- **谁**：客户经理
- **何时**：每周
- **如何**：
  1. 系统识别缺失地市、负责人、客户等级、关键人、最近互动的客户。
  2. 客户经理每周补齐字段或提交无法补齐原因。
  3. 区域负责人跟进未处理项。
- **产出**：已补充的客户数据

#### Step 4：月度区域复盘

- **谁**：区域负责人
- **何时**：每月最后一周
- **如何**：
  1. 查看地市覆盖、重点客户、预警客户、有效商机、拜访达成、数据质量。
  2. 对重点客户覆盖不足、长期未互动、低质量数据或无人负责客户，指定处理人和完成时间。
  3. 导出月度复盘数据（当前为手动统计，后续将支持报表导出）。
  4. 作为下月客户经营计划输入。
- **产出**：月度复盘报告，下月行动计划

### 检查清单

- [ ] 客户已标注地市、行业、等级和负责人
- [ ] 重点客户已进入重点经营清单
- [ ] 低数据质量客户已有补充计划
- [ ] 90 天未互动重点客户已分配维护动作
- [ ] 月度区域复盘已导出
- [ ] 下月经营计划已制定

### 异常处理

| 异常场景 | 处理方式 | 升级路径 |
|---|---|---|
| 客户归属争议 | 两个客户经理声称同一客户 | 区域负责人按客户实际服务历史裁定 → 主管备案 |
| 地市数据错误 | 客户实际所在地与系统不符 | 客户经理修正 → 区域负责人审核 |
| 重点客户无负责人 | 客户重要性高但无人跟进 | 区域负责人立即指派 → 主管确认 |
| 数据质量长期低于 60 分 | 字段缺失严重 | 客户经理限期补齐 → 区域负责人跟进 → 主管评估 |
| 区域覆盖空白 | 某地市无客户或客户极少 | 区域负责人制定拓展计划 → 主管审批资源 |

## SOP-10 客户拜访计划与互动回填流程

| 元数据 | 值 |
|---|---|
| **版本** | 1.1 |
| **负责人** | 客户经理/CSM |
| **触发条件** | 重点客户维护、商机跟进、项目关键节点、培训前后、客户风险预警 |
| **预计用时** | 拜访计划 3-5 分钟；互动记录 5-10 分钟 |
| **审核周期** | 每周 |

### 目的

把客户拜访和日常互动从个人记忆转为系统记录，确保拜访有目的、互动有结果、下一步有提醒。

### 范围

- **适用**：客户经理/CSM、项目经理、运维工程师、培训负责人、主管
- **时机**：重点客户维护、商机跟进、项目关键节点、培训前后、客户风险预警
- **不包含**：非计划内的临时电话（可后续补录）

### RACI 矩阵

| 步骤 | 执行者（R） | 决策者（A） | 咨询方（C） | 知会方（I） |
|---|---|---|---|---|
| 创建拜访计划 | 客户经理 | 客户经理 | 项目经理 | 客户 |
| 准备客户简报 | 客户经理 | 客户经理 | 运维工程师 | — |
| 执行拜访 | 客户经理 | 客户经理 | — | 客户 |
| 现场签到 | 客户经理 | 系统 | — | 主管 |
| 回填互动记录 | 客户经理 | 客户经理 | — | 团队 |
| 超时提醒 | 系统 | 主管 | — | 客户经理 |

### 流程图

```
[确定拜访需求]
    ├─→ 重点客户维护周期
    ├─→ 商机跟进需要
    ├─→ 项目关键节点（验收/上线/培训）
    ├─→ 培训前后
    └─→ 客户风险预警
    ↓
[创建拜访计划]
    ├─→ 客户（必填）
    ├─→ 拜访日期（必填）
    ├─→ 拜访目的（必填）
    ├─→ 拜访地点（必填）
    ├─→ 预期结果（可选）
    └─→ 关联商机/项目/培训（可选）
    ↓
[状态：计划中]
    ↓
[系统提醒准备]
    ├─→ 客户简报（客户概况/项目/资产/近期问题/商机/关键人/互动）
    ├─→ 方案材料
    └─→ 项目资料
    ↓
[执行拜访]
    ↓
[现场签到]（点击「签到」按钮，记录签到时间）
    ↓
[状态变为：已完成]
    ↓
[48 小时内回填互动记录]
    ├─→ 互动类型（现场/电话/微信/会议/培训/邮件/其他）
    ├─→ 互动对象
    ├─→ 关键内容
    ├─→ 客户反馈
    ├─→ 下一步动作
    └─→ 下次跟进时间
    ↓
[如产生新需求/商机/风险]
    ├─→ 创建商机信号
    ├─→ 创建任务
    └─→ 登记风险
    ↓
[系统更新]
    ├─→ 客户最近互动时间
    ├─→ 关系亲疏度
    ├─→ 客户详情页时间线
    ↓
[逾期未回填（48h）]
    ├─→ 提醒负责人
    └─→ 超过 72h → 升级给主管
```

### 详细步骤

#### Step 1：创建拜访计划

- **谁**：客户经理
- **何时**：确定拜访需求后
- **如何**：
  1. 进入「商机营销 → 拜访计划」，点击「新建拜访」。
  2. 填写客户（必填）、拜访日期（必填）、拜访目的（必填）、拜访地点（必填）、预期结果（可选）。
  3. 如拜访与商机、项目、培训相关，关联对应记录。
  4. 状态为「计划中」。
  5. 保存。
- **产出**：拜访计划，状态「计划中」

#### Step 2：准备客户简报

- **谁**：客户经理
- **何时**：拜访前 1-2 天
- **如何**：
  1. 查看客户档案（概况、项目、资产、近期问题、商机、关键人、最近互动）。
  2. 准备方案材料或项目资料。
  3. 确认拜访对象和目的。
- **产出**：客户简报，准备材料

#### Step 3：执行拜访并签到

- **谁**：客户经理
- **何时**：拜访当天
- **如何**：
  1. 到达拜访地点后，打开系统，点击「签到」。
  2. 确认签到后，系统记录签到时间。
  3. 执行拜访。
  4. 拜访结束后，手动将状态标记为「已完成」（或系统自动根据签到标记）。
- **产出**：已签到拜访记录，状态「已完成」

#### Step 4：回填互动记录

- **谁**：客户经理
- **何时**：拜访完成后 48 小时内
- **如何**：
  1. 进入「商机营销 → 营销跟进」，点击「新增跟进」。
  2. 选择客户、关联商机（可选）、跟进类型（电话/邮件/拜访/会议/其他）。
  3. 填写跟进内容（必填）、下一步计划（可选）。
  4. 保存。
  5. 如产生新需求/商机/风险，创建商机信号或任务。
- **产出**：互动记录，可能的商机/任务/风险

### 检查清单

- [ ] 拜访计划已明确目的和对象
- [ ] 已准备客户简报或方案材料
- [ ] 已现场签到（记录时间）
- [ ] 互动后 48 小时内已回填记录
- [ ] 下一步动作和下次跟进时间已填写
- [ ] 新需求或风险已转为商机、任务或风险记录
- [ ] 客户最近互动时间已更新

### 异常处理

| 异常场景 | 处理方式 | 升级路径 |
|---|---|---|
| 客户临时取消拜访 | 记录取消原因 | 系统标记取消 → 重新安排 |
| 拜访后未签到 | 忘记签到 | 手动补录签到时间 → 说明原因 |
| 48 小时内未回填 | 工作繁忙 | 系统提醒 → 超过 72h 升级给主管 |
| 互动记录内容过于简单 | 无法复盘 | 主管要求补充 → 退回修改 |
| 拜访中发现重大商机/风险 | 需要紧急处理 | 立即创建商机/风险 → 主管同步 |
| 客户拒绝后续跟进 | 客户态度明确 | 标记放弃并说明原因 |

## SOP-11 权力地图与客户关系维护流程

| 元数据 | 值 |
|---|---|
| **版本** | 1.1 |
| **负责人** | 客户经理/CSM |
| **触发条件** | 新增重点客户、项目启动、商机确认、客户组织变化、客户交接 |
| **预计用时** | 初始维护 30-60 分钟；月度检查 15-30 分钟 |
| **审核周期** | 每月 |

### 目的

沉淀客户内部决策链、跨客户关系和团队协作过程，降低人员离职、客户经理变更或项目交接造成的关系资产流失。

### 范围

- **适用**：客户经理/CSM、区域负责人、主管、项目经理、运维负责人
- **时机**：新增重点客户、项目启动、商机确认、客户组织变化、客户交接
- **不包含**：非业务相关的个人隐私信息（家庭、爱好等）

### RACI 矩阵

| 步骤 | 执行者（R） | 决策者（A） | 咨询方（C） | 知会方（I） |
|---|---|---|---|---|
| 新增关键人 | 客户经理 | 客户经理 | 项目经理 | 区域负责人 |
| 维护权力地图 | 客户经理 | 客户经理 | 运维工程师 | 团队 |
| 维护客户树 | 客户经理 | 区域负责人 | 主管 | 团队 |
| 添加协作评论 | 团队成员 | 客户经理 | — | 相关成员 |
| 客户交接 | 接手人 | 主管 | 原负责人 | 区域负责人 |

### 流程图

```
[新增重点客户/项目启动/商机确认/组织变化]
    ↓
[进入客户档案 → 客户详情页]
    ↓
[新增关键人]
    ├─→ 姓名、职务、部门
    ├─→ 角色类型：决策人/影响人/使用人/信息人/反对者/其他
    ├─→ 权力等级、影响力（0-100）、亲疏度、立场、决策角色
    ├─→ 关键诉求、联系方式
    └─→ 不记录无业务必要的隐私信息
    ↓
[在权力地图中维护]
    ├─→ 关键人节点
    ├─→ 上下级关系
    ├─→ 影响路径
    ├─→ 支持/反对立场
    └─→ 决策链路
    ↓
[在客户树中维护]
    ├─→ 集团、上下级单位
    ├─→ 关联单位、供应链
    ├─→ 竞争关系
    └─→ 人事流动
    ↓
[团队协作]
    ├─→ 评论、标注
    ├─→ 协作任务
    └─→ @ 成员
    ↓
[系统检查完整性]
    ├─→ 重点客户至少 1 个决策人 + 1 个影响人/使用人
    └─→ 缺少则提示「关系链不完整」
    ↓
[客户交接]
    ├─→ 接手人查看权力地图、客户树、协作任务、最近互动、商机、风险
    ├─→ 确认接收
    └─→ 更新操作日志
```

### 详细步骤

#### Step 1：新增关键人

- **谁**：客户经理
- **何时**：新增重点客户或发现新客户关键人时
- **如何**：
  1. 进入「客户资产 → 客户档案」，找到客户，点击「编辑」或「新增关键人」（当前通过客户详情页维护）。
  2. 进入「客户资产 → 关系网络」，点击「新增关键人」。

  4. 填写关系（直接/间接）、影响力（0-100）、联系方式（必填）。
  5. 保存。
- **产出**：关键人记录，进入关系网络列表

#### Step 2：维护权力地图（当前为列表形式，后续将可视化）

- **谁**：客户经理
- **何时**：关键人信息变化时

  1. 在「关系网络」中查看关键人列表。

#### Step 3：维护客户树

#### Step 4：团队协作与评论

- **如何**：
  1. 在客户详情页或关系网络中添加评论。

  3. 创建协作任务。
- **产出**：协作评论，任务

#### Step 5：客户交接

- **谁**：接手人
- **何时**：客户经理变更时
- **如何**：
  1. 接手人查看权力地图、客户树、协作任务、最近互动、商机、风险。
  2. 确认接收。
  3. 系统更新操作日志。
- **产出**：交接完成，日志记录

## SOP-12 巡检计划执行与记录流程

| 元数据 | 值 |
|---|---|
| **版本** | 1.0（新增） |
| **负责人** | 运维负责人 |
| **触发条件** | 按巡检计划周期执行（每日/每周/每月/每季度） |
| **预计用时** | 单次巡检 15-60 分钟；记录 5-10 分钟 |
| **审核周期** | 每周 |

### 目的

确保巡检计划按周期执行，巡检结果完整记录，异常情况及时高亮，形成可追溯的运维资产。

### 范围

- **适用**：运维工程师、运维负责人
- **时机**：按巡检计划周期执行
- **不包含**：非计划内的临时检查（可转为运维记录）

### RACI 矩阵

| 步骤 | 执行者（R） | 决策者（A） | 咨询方（C） | 知会方（I） |
|---|---|---|---|---|
| 创建巡检计划 | 运维负责人 | 运维负责人 | 项目经理 | 客户 |
| 执行巡检 | 运维工程师 | 运维工程师 | — | 运维负责人 |
| 记录巡检结果 | 运维工程师 | 运维工程师 | — | 系统 |
| 处理异常 | 运维工程师 | 运维负责人 | 客户经理 | 客户 |
| 审核巡检记录 | 运维负责人 | 运维负责人 | — | 主管 |

### 流程图

```
[创建巡检计划]
    ├─→ 计划名称、类型（日常/专项/应急）
    ├─→ 频率（每日/每周/每月/每季度）
    ├─→ 执行人
    ├─→ 检查项（至少 1 项）
    │       ├─→ 检查项名称
    │       └─→ 检查标准
    └─→ 状态：启用/暂停
    ↓
[系统按日历展示计划]
    ├─→ 列表视图
    └─→ 日历视图（7 列网格，周日-周六）
    ↓
[执行日期到达]
    ↓
[运维工程师执行巡检]
    ├─→ 按检查项逐项检查
    ├─→ 记录结果：正常/异常/部分异常
    └─→ 异常项详细说明
    ↓
[保存巡检记录]
    ├─→ 异常记录红色高亮
    ├─→ 自动关联巡检计划（后续将自动关联）
    └─→ 通知运维负责人
    ↓
[处理异常]
    ├─→ 轻微异常：记录并观察
    ├─→ 严重异常：创建运维记录 → 可能触发商机信号
    └─→ 紧急异常：立即升级 → 客户通知
    ↓
[运维负责人审核]
    ├─→ 确认记录完整性
    └─→ 确认异常已处理
```

### 详细步骤

#### Step 1：创建巡检计划

- **谁**：运维负责人
- **何时**：新客户接入、合同签署、定期维护安排时
- **如何**：
  1. 进入「运维管理 → 巡检计划」，点击「新建计划」。
  2. 填写计划名称（必填）、类型（日常/专项/应急）、频率（每日/每周/每月/每季度）、周期（Input）、执行人（必填）。
  3. 添加检查项（至少 1 项）：检查项名称 + 检查标准。
  4. 状态默认「启用」。
  5. 保存。
- **产出**：巡检计划，按日历展示

#### Step 2：执行巡检

- **谁**：运维工程师
- **何时**：按巡检计划日期
- **如何**：
  1. 查看日历视图或列表视图，找到当天计划。
  2. 按检查项逐项执行巡检。
  3. 记录每项结果：正常/异常/部分异常。
  4. 异常项详细说明。
- **产出**：巡检结果

#### Step 3：记录巡检结果

- **谁**：运维工程师
- **何时**：巡检完成后
- **如何**：
  1. 进入「运维管理 → 巡检记录」。
  2. 填写巡检时间、计划名称、执行人、检查项数、异常数、结果。
  3. 异常记录红色高亮。
  4. 保存。
  5. 当前为独立记录，后续将支持巡检计划自动关联生成巡检记录。
- **产出**：巡检记录，异常高亮

#### Step 4：处理异常

- **谁**：运维工程师 / 运维负责人
- **何时**：巡检发现异常时
- **如何**：
  1. 轻微异常：记录并观察。
  2. 严重异常：创建运维记录，可能触发商机信号（SOP-01→SOP-02）。
  3. 紧急异常：立即升级给客户和主管。
- **产出**：异常处理记录，可能的商机信号

### 检查清单

- [ ] 巡检计划已创建并启用
- [ ] 检查项至少 1 项，标准明确
- [ ] 已按周期执行巡检
- [ ] 巡检结果已记录（正常/异常）
- [ ] 异常项已详细说明并高亮
- [ ] 严重异常已创建运维记录
- [ ] 运维负责人已审核记录完整性

### 异常处理

| 异常场景 | 处理方式 | 升级路径 |
|---|---|---|
| 巡检计划遗忘执行 | 工作繁忙或计划不明确 | 系统提醒 → 运维负责人督促 |
| 检查项不适用 | 客户环境变化 | 调整检查项 → 运维负责人审核 |
| 巡检结果全部异常 | 系统/环境重大故障 | 立即创建运维记录 → 升级主管和客户 |
| 巡检记录与实际情况不符 | 记录错误或遗漏 | 重新巡检 → 修正记录 |
| 客户拒绝巡检 | 客户不配合 | 记录原因 → 客户经理协调 → 主管 |

## SOP-13 数据导出与报表使用流程

| 元数据 | 值 |
|---|---|
| **版本** | 1.0（新增） |
| **负责人** | 区域负责人 / 主管 / 项目经理 |
| **触发条件** | 月度复盘、季度汇报、客户经营分析、数据备份 |
| **预计用时** | 配置导出 2-5 分钟；等待处理视数据量 |
| **审核周期** | 按需 |

### 目的

支持业务数据按需求导出，便于离线分析、汇报和备份，同时保留导出历史便于审计。

### 范围

- **适用**：区域负责人、主管、项目经理、客户经理、运维负责人
- **时机**：月度复盘、季度汇报、客户经营分析、数据备份
- **不包含**：实时数据查询（在线查看即可）

### RACI 矩阵

| 步骤 | 执行者（R） | 决策者（A） | 咨询方（C） | 知会方（I） |
|---|---|---|---|---|
| 配置导出 | 业务人员 | 业务人员 | — | 系统 |
| 系统生成文件 | 系统 | 系统 | — | 业务人员 |
| 下载文件 | 业务人员 | 业务人员 | — | — |
| 查看导出历史 | 业务人员 | 业务人员 | — | 管理员 |
| 审计导出记录 | 管理员 | 主管 | — | — |

### 流程图

```
[确定导出需求]
    ├─→ 月度复盘数据
    ├─→ 季度汇报报表
    ├─→ 客户经营分析
    └─→ 数据备份
    ↓
[进入作战台 → 数据导出]
    ↓
[配置导出参数]
    ├─→ 数据类型：项目/客户/工单/财务
    ├─→ 时间范围（RangePicker）
    └─→ 格式：Excel（绿色）/ PDF（红色）
    ↓
[点击「开始导出」]
    ↓
[系统处理]
    ├─→ 处理中（蓝色状态标签）
    └─→ 已完成（绿色状态标签）
    ↓
[下载文件]
    ├─→ Excel：表格数据
    └─→ PDF：格式化报表
    ↓
[查看导出历史]
    ├─→ 导出类型、时间范围、格式、状态、大小、操作人、创建时间
    └─→ 仅「已完成」状态显示下载链接
    ↓
[当前为前端模拟，后续将接入后端真实导出]
```

### 详细步骤

#### Step 1：配置导出

- **谁**：业务人员
- **何时**：有导出需求时
- **如何**：
  1. 进入「作战台 → 数据导出」。
  2. 选择数据类型（项目/客户/工单/财务）。
  3. 选择时间范围（RangePicker）。
  4. 选择格式：Excel（绿色）或 PDF（红色）。
  5. 点击「开始导出」。
- **产出**：导出任务，状态「处理中」

#### Step 2：下载文件

- **谁**：业务人员
- **何时**：导出状态变为「已完成」
- **如何**：
  1. 在导出历史列表中找到已完成任务。
  2. 点击「下载」链接。
  3. 保存文件到本地。
- **产出**：Excel 或 PDF 文件

#### Step 3：查看导出历史

- **谁**：业务人员 / 管理员
- **何时**：需要审计或查找历史导出时
- **如何**：
  1. 在「数据导出」页查看导出历史表格。
  2. 列：导出类型、时间范围、格式标签、状态标签、大小、操作人、创建时间、操作。
  3. 仅「已完成」显示下载链接。
- **产出**：导出历史记录

### 检查清单

- [ ] 已选择正确的数据类型
- [ ] 已选择正确的时间范围
- [ ] 已选择正确的格式（Excel/PDF）
- [ ] 导出任务已完成
- [ ] 已下载文件并验证内容
- [ ] 敏感数据已脱敏（客户联系方式、金额等）

### 异常处理

| 异常场景 | 处理方式 | 升级路径 |
|---|---|---|
| 导出任务失败 | 数据量大或系统故障 | 重新导出 → 管理员检查系统状态 |
| 导出数据不完整 | 筛选条件不当 | 调整时间范围 → 重新导出 |
| 导出文件无法打开 | 格式错误或文件损坏 | 重新导出 → 如仍失败联系管理员 |
| 导出敏感数据 | 隐私/合规风险 | 管理员审核导出权限 → 敏感字段脱敏 |
| 导出历史丢失 | 系统问题 | 联系管理员 → 检查日志 |
| 数据量过大导致超时 | 超出系统处理能力 | 分批导出 → 缩小时间范围 → 管理员优化 |

### 23.3 技术实现说明

**前端技术栈**：React 18 + TypeScript + Vite 5 + Ant Design 5 + React Router 6 + Zustand + Axios。

**前端目录**：
```
src/
  App.tsx          # 路由配置（懒加载）
  main.tsx         # 入口
  components/
    MainLayout.tsx # 主布局（侧边栏+顶部栏+内容区）
  pages/
    Login.tsx      # 登录页
    cockpit/       # 作战台（Overview, Report, Export, Team）
    project/       # 项目交付（Overview, Nodes, Meeting, Kanban）
    ops/           # 运维管理（Records, Plan, Log, Assets）
    biz/           # 商机营销（Pool, Follow, Visit）
    customer/      # 客户资产（Assets, Map, Network）
    knowledge/     # 知识分享（Training, Material）
  store/
    auth.ts        # 认证状态（Zustand）
  utils/
    request.ts     # Axios封装（含拦截器）
```

**后端技术栈**：NestJS 10 + Prisma 5 + PostgreSQL + Redis + MinIO + JWT + Swagger。

**后端API**：`/auth/*`、`/users/*`、`/dashboard/*`、`/projects/*`、`/ops/*`、`/biz/*`、`/customers/*`、`/knowledge/*`、`/notifications/*`、`/upload/*`、`/health`。
